# 开发文档-⑦服务端中心化数据与文件同步（Remote-First 优化版）

## 1. 核心设计变更 (Remote-First)
### 1.1 现状 (Local-Centric)
*   **模式**：客户端是“加工厂”，服务器是“搬运工”。
*   **痛点**：本地修改需全量同步回传，多端登录易冲突，数据一致性维护成本高。

### 1.2 目标 (Remote-First)
*   **服务器 (Gateway)**：作为**“执行与状态中心”**。
    *   **主存储**：持有所有批次的 Excel 和图纸。
    *   **主执行**：直接在服务器端修改 Excel 内容、记录异常、执行提交。
*   **客户端 (App)**：作为**“展示与指令终端”**。
    *   **指令化操作**：填报价格、挂起异常等操作通过 API 立即通知服务器执行。
    *   **本地镜像**：仅下载服务器数据用于本地高速查看和图纸预览。

---

## 2. 存储位置与配置定义

### 2.1 后端服务器存储 (Server Storage)
1.  **文件中心**：配置 `Storage:ServerRoot`，存放批次文件夹。
2.  **状态中心**：PostgreSQL 记录批次生命周期（Todo -> Done）及异常池。
3.  **操作原子化**：后端引入 `NPOI` 库，实现不依赖 Office 即可直接读写服务器磁盘上的 Excel 文件。

### 2.2 客户端本地存储 (Local Storage)
*   **定位**：只读镜像缓存。
*   **逻辑**：维持现有目录结构，但本地 Excel 仅用于 UI 绑定展示。用户点击“保存”时，优先发 API 给后端，后端成功后再更新本地缓存。

---

## 3. 服务端（TZHJ.Gateway）改造方案

### 3.1 核心 API 重新定义
1.  **`GET /api/sync/catalog` (同步清单)**
    *   下发当前用户所有批次的元数据及处理状态。
2.  **`POST /api/batch/update-row` (行数据更新) - [新增]**
    *   **输入**：批次号、行 Key、修改的字段键值对。
    *   **动作**：服务器直接打开磁盘上的 `Data.xlsx`，定位并修改对应单元格，保存文件。
3.  **`POST /api/batch/suspend-exception` (挂起异常) - [新增]**
    *   **动作**：直接在数据库 `ExceptionPool` 插入记录。
4.  **`POST /api/batch/submit` (批次提交)**
    *   **动作**：服务器直接读取自己磁盘上已修改好的 Excel，执行业务推送（EBS/SRM），成功后移动文件夹到 `Done`。

---

## 4. 客户端（TZHJ.App）改造方案

### 4.1 操作逻辑重构 (指令化)
客户端不再直接修改本地文件，而是将所有意图通过 API 发送给后端执行：

| 客户端操作场景 | 对应 API 动作 | 后端处理逻辑 |
| :--- | :--- | :--- |
| **填报业务数据** (填价格/选是或否) | `POST /api/batch/update-row` | 利用 NPOI 直接修改服务器磁盘 Excel 对应单元格 |
| **挂起异常** (点击异常按钮) | `POST /api/batch/suspend-exception` | 在 DB 记录异常单，并将批次内该行标记为异常状态 |
| **补处理/撤销异常** | `POST /api/batch/resolve-exception` | 更新 DB 状态，同步修改 Excel 记录 |
| **最终提交回传** | `POST /api/batch/submit` | 读取最新 Excel 执行业务推送，成功后移动文件夹至 Done |
| **手动/自动获取新数据** | `POST /api/sync/trigger-fetch` | 后端执行取数逻辑并落地服务器磁盘，更新 Catalog |

### 4.2 本地镜像联动
*   **写后读同步**：客户端在收到 API 成功响应后，应立即触发该批次的单项同步，确保本地镜像 Excel 与服务器保持一致。
*   **静默轮询**：后台定期拉取 `Catalog`，若发现服务器上某批次的 `LastModified` 晚于本地，则静默更新本地副本。

### 4.3 覆盖范围声明 (Mandatory)
本方案要求客户端**严禁**直接通过文件 I/O 改变任何业务数据（Excel 内容）或物理结构（移动文件夹）。
1. 所有导致单元格变动的操作，必须走 API。
2. 所有导致批次状态流转（待处理 -> 已处理）的操作，必须走 API。
3. 所有异常池的增删改，必须走 API。
客户端的本地 I/O 权限仅限于“创建镜像”和“读取展示”。

---

## 5. 关键技术细节
1.  **并发锁**：后端修改 Excel 时需处理好文件读写锁，防止多用户（或同一用户多端）同时修改同一个 Excel 导致损坏。
2.  **事务一致性**：对于异常挂起等操作，需确保数据库记录更新与业务状态流转的原子性。

---

## 6. 实施路线
1.  **Step 1**: 在 `TZHJ.Gateway` 实现后端 Excel 读写组件及数据库表结构扩展。
2.  **Step 2**: 实现行级更新 API (`update-row`) 及异常池 API。
3.  **Step 3**: 改造客户端，将原本的“本地保存”动作改为“API 指令优先+本地同步”。
4.  **Step 4**: 实现服务器端直接触发业务提交的逻辑。
