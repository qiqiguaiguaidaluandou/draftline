# 开发文档-⑧：基于组织架构的精细化权限与目录重构方案（Review 增强版）

## 1. 业务逻辑定义与目标
本方案旨在通过引入“业务组（Group）”维度，实现同一次取数动作下的内容分流与精细化权限管理。
*   **组 (Group)**：内容分流标识。同一次采集可产生多个 Group 数据，由不同权限的人处理。
*   **权限原则**：基于工号的 **“视野隔离”**。默认无权，按表授权，组内共享。

---

## 2. 统一存储目录标准

### 2.1 客户端本地结构 (镜像化)
层级逻辑：**根目录 > 流程 > 状态 > 业务组 > 批次号**
1.  **图纸核价**：
    `我的文档\data\图纸核价\{待处理 | 已处理}\{组名}\{批次文件夹}\`
2.  **机加中心挑图**：
    `我的文档\data\机加中心挑图\{待处理 | 已处理}\{Default | 组名}\{批次文件夹}\`

### 2.2 服务器端结构 (扁平化)
层级逻辑：**根目录 > 流程 > 业务组 > 批次号**
*   **规则**：服务器磁盘不包含“状态”层级，不物理搬运文件夹。

---

## 3. 精细化管理逻辑 (增强项)

### 3.1 权限白名单机制 (Security)
*   **Deny All**：数据库 `UserPermissions` 未定义的工号，登录后无法获取任何数据。
*   **多维绑定**：权限精确到 `(EmployeeId, Flow, GroupName)`。

### 3.2 客户端数据生命周期 (Retention)
*   **待处理 (Todo) 镜像**：全量镜像。只要服务器状态为待处理且在权限范围内，本地必须保持同步，无时间限制，确保工作不遗漏。
*   **已处理 (Done) 镜像**：15天滚动镜像。客户端仅自动同步/保留最近 15 天内转为“已处理”的批次。
*   **过期清理**：超过 15 天的历史“已处理”镜像，将由客户端定期触发静默清理，释放本地磁盘空间。

### 3.3 组内协作异常池 (Collaboration)
*   **归属变更**：异常不再属于个人，而是属于 **“业务组”**。
*   **共享视野**：同组内的任何成员都可以看到并补处理该组下的异常条目。

---

## 4. 数据库与 API 变更设计

### 4.1 字段扩展
*   **`BatchRegistry`** & **`Exceptions`** 表：统一增加 `GroupName` 字段。
*   **联合索引**：`Unique(Flow, GroupName, BatchId)` 确保唯一性。

### 4.2 API 原子指令增强
所有的写指令 API 必须强制包含 `GroupName` 以实现精确定位：
*   `POST /api/batch/update-row` (Flow, **GroupName**, BatchId, ...)
*   `POST /api/batch/suspend-exception` (Flow, **GroupName**, BatchId, ...)
