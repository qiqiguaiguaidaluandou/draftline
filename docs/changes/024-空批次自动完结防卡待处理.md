# 024 空批次（0 行）自动完结，防止永远卡在"待处理"

- 状态：**已实现，待真机验证**
- 日期：2026-07-10
- 关联提交：64f4800（实现）

> 结论：取数时某个 (流程/组/窗口) 取到 **0 行**的空批次，仍照旧落盘 + 登记（推进 023 的 `T_covered`
> 高水位、防重采），但**新建时直接置终态 `Done` + 哨兵 `AuditId = "AUTO-EMPTY-{batchId}"`**，
> 从源头避免"无行可提交却一直停在待处理"。改动仅一处（`PersistGroupAsync`），无迁移、无新表、无客户端改动。

## 需求 / 改动（原始问题）

目前的逻辑是：某批次即使**没有取到任何数据**也会形成批次文件——只是没有数据。该空批次以
`Status = Todo, TotalRows = 0` 落库并下发到客户端，落在**待处理**列表里。但空批次**没有行可提交**
（客户端提交闸门 `BatchWorkViewModel.CanSubmit` 要求 `Rows.Count > 0`），于是它**永远停在待处理**，
只能等 023/022 的保留期清理（`RetentionDaysForTodo`，默认 30 天）到期才被物理删除。

**期望**：空批次不该占着"待处理"等一个永远不会发生的提交动作，应从源头收口为终态，同时不破坏
023 的取数正确性（不能因此重采、不能丢水位）。

## 现状根因（为什么空批次必须登记）

`DataIngestionService.PersistGroupAsync` 有一处刻意设计（改前注释）：

> `// 注意：不再因 rows.Count==0 早退——空组也要落盘(文件夹+空表)并登记 TotalRows=0。`

原因是 023 的**数据时间轴高水位 `T_covered`** 是**从已登记批次派生**的（`ComputeTCoveredAsync` →
`IngestionPlanner.HighWatermark`）。若空窗不登记：

- 该窗口永远"未覆盖"（`窗口止 > T_covered` 恒成立）→ 每轮 `Plan` 都重新规划它 → 反复调 EBS 取 0 行 →
  **无限重采**。

所以"空窗也要登记"是**防重采/推进水位的必要条件**，不能删。问题只在于登记后它的**状态**不该是 `Todo`。

**核心矛盾**：登记必须保留（为水位线），但它不该出现在"待处理"。正确方向是——**登记，但以终态落库**，
而不是不登记。

## 方案

在 `PersistGroupAsync` 里，当该组本窗口 `rows.Count == 0` 时：

- 仍照旧落盘（文件夹 + 空表）与登记 `TotalRows = 0`（水位、防重采不变）；
- **新建批次**时状态直接置 `BatchLocation.Done`，并写哨兵 `AuditId = "AUTO-EMPTY-{batchId}"`；
- 另写一条 `Status = "EmptyAutoDone"` 的 `ActivityLog` 留痕，审计上可区分"空窗自动完结"与"用户回传完成"。

关键决策与取舍：

1. **复用 `Done` + 哨兵 `AuditId`，不新增枚举值。** `BatchLocation` 仍只有 `Todo/Done`，改动最小、无需动
   客户端渲染与 DB 迁移。代价是 `Done` 里混入了"空窗自动完结"与"用户真实回传完成"两种含义——靠 `AuditId`
   前缀区分（真实回传形如 `AUDIT-...`，见 `/submit`；空窗为 `AUTO-EMPTY-...`）。
   （备选"新增 `Empty` 终态"审计语义更清晰，但要改枚举 + 客户端渲染 + DB 多处，已权衡后不采用。）
2. **置 `Done` 不影响水位。** `HighWatermark` 只看批次 ID（窗口止），不看状态，故自动完结不改变 `T_covered`
   的推进；空批次照常"算采过"。
3. **服务端 `Done` 只是 DB 标志。** `/submit` 置 `Done` 时并无物理搬移，故此处置 `Done` 同样只是登记状态，
   不做文件夹移动。客户端按 `BatchLocation` 分组，空批次将落在**已处理**页而非待处理页。
4. **磁盘自愈重存分支加保护。** 当"数据库有登记但磁盘丢失"触发自愈重取（走 `else` 分支）、且这次恰好取到
   0 行时，**只有仍是 `Todo` 且 `AuditId == null`（从没被用户处理过）的批次**才收口为 `Done`；
   **已提交/已处理（`Done` 或已有真实 `AuditId`）的批次不回退、不覆盖审计号**，避免篡改已完成批次的审计链。

## 实现（已落地，blast radius 极小）

- `src/Draftline.Gateway/DataIngestionService.cs` · `PersistGroupAsync`：
  - 计算 `isEmpty = rows.Count == 0` 与 `emptyAuditId = isEmpty ? "AUTO-EMPTY-{batchId}" : null`；
  - **新建分支**（`registry == null`）：`Status = isEmpty ? Done : Todo`，`AuditId = emptyAuditId`；
  - **自愈分支**（`else`）：仅当 `isEmpty && Status == Todo && AuditId == null` 时收口为 `Done` + 哨兵，
    否则保持原状态与审计号不动；
  - `ActivityLog` 的 `Status` 在空窗时记 `"EmptyAutoDone"`（否则 `"Success"`），并带上 `emptyAuditId`；
  - 更新了那处 `rows.Count==0` 的注释以反映新行为。
- **无数据库迁移、无新模型、无新表、无枚举改动、无客户端改动。**

有数据的批次行为完全不变（照旧 `Todo`、`AuditId = null`）。

  - 核价某组空、另一组有数据 → 空组自动完结、有数据组仍进待处理；
  - 已提交批次磁盘丢失触发自愈、且该窗此时取到 0 行 → 状态与真实 `AuditId` 不被覆盖。

## 备注 / 待办

- **既有旧空批次不追溯**：本次修复前已生成的 `Todo` 空批次（磁盘+登记都在）会被取数预检直接跳过，
  不会走到新逻辑，仍按原有 30 天保留期（`RetentionDaysForTodo`）自然清理。如需立即清掉，可另做一次性回填
  （把 `TotalRows==0 && Status==Todo && AuditId==null` 的批次批量置 `Done`）——按需再定。
- **可选增强**：若希望空批次连"已处理"页也不显示，可在 `ApiEndpoints` 的 `/sync/catalog` 追加
  `TotalRows > 0` 过滤（独立小改动，本次未做）。
- 关联 [023] 的高水位模型：空批次仍作为水位证据参与 `T_covered` 派生，本改动不影响其正确性。
