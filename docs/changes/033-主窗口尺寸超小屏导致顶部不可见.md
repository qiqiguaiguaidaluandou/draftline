# 033 修复主窗口尺寸超出小屏导致顶部标题栏不可见

- 状态：**实现完成，待运行态验证**
- 日期：2026-07-17
- 关联提交：`__PENDING__`

## 需求 / 现象

登录进入软件后，主窗口"最顶部直接看不见"，整体尺寸不合理、无法正常拖动。

## 根因

`ShellWindow`（登录后的主窗口）用固定 `Height=800 Width=1220` + `WindowStartupLocation=CenterScreen`，且没有对屏幕工作区做任何裁剪。

在小屏笔记本（如 1366×768，扣除任务栏后工作区高约 728）上，800 的高度放不下；`CenterScreen` 按窗口自身高度垂直居中时会把 `Top` 算成负值，于是**窗口顶部（含系统标题栏）被推到屏幕上边界之外**——看不见，也无法拖动。大屏因放得下而正常，故此前未暴露。

## 方案

**最小改动，只修主窗口的启动尺寸与定位**，不动任何业务逻辑：

- XAML：定位 `CenterScreen` → `Manual`，补最小尺寸 `MinHeight=560 MinWidth=900`；默认 800×1220 保留。
- 代码后端：`SourceInitialized`（布局前）调用新增 `FitToWorkArea()`：
  - 把宽/高裁剪到当前屏幕工作区以内（留 8px 边距，且不低于最小尺寸）；
  - 在工作区内居中，并夹住左/上边界——保证标题栏永远可见、可拖动。
- 大屏行为不变（仍是 800×1220 居中）；小屏才自动缩小适配。

`LoginWindow` 为 600×520，基本不会溢出，未改动。

## 实现

- `src/Draftline.App/Views/ShellWindow.xaml`：`WindowStartupLocation=Manual` + `MinHeight/MinWidth`。
- `src/Draftline.App/Views/ShellWindow.xaml.cs`：新增 `FitToWorkArea()`，`SourceInitialized` 时调用。

## 验证

- `dotnet build src/Draftline.App -p:EnableWindowsTargeting=true`：**编译通过**。
- 运行态待真机确认（尤其那台出问题的小屏机器）：登录后主窗口应完整显示、标题栏可见可拖动，且不超出屏幕工作区。
