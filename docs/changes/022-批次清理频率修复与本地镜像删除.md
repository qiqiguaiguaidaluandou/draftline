# 022 批次清理频率修复与本地镜像删除

- 状态：已完成
- 日期：2026-07-08
- 关联提交：<待提交时补>

## 需求 / 改动

两处围绕"批次生命周期"的问题:

1. **后端过期清理触发不可靠（bug）**：`DataIngestionService` 用 `if (DateTime.Now.Minute % 60 == 0)` 想做到"每小时清理一次"。但该条件等价于"分钟数恰好等于 0"，而 5 分钟循环用 `Task.Delay` 不对齐时钟、会漂移，某些启动时刻的 tick 永远命中不了 minute==0 → 周期清理长期不触发，实际只剩"每次重启清一次"。
2. **服务器过期清理后，客户端本地副本不会同步删除**：镜像同步 `MirrorSyncAsync` 只做"补齐/更新/移动"，没有"删除本地存在但云端已无"的一步。服务器按保留期（Done 15 天 / Todo 30 天）删掉批次后，操作员 PC 上对应文件夹会一直残留成孤儿，越积越多。期望改成"服务器删了 → 本地也删"，保持"本地 = 云端镜像"。

## 方案

### 1. 清理频率：改按经过时间触发

不再依赖 tick 是否落在整点分钟，改为记录 `lastCleanup`、判断"距上次清理是否已满 `cleanupInterval`"。间隔取 **6 小时**（一天约 4 次）——过期判定按"天"计，无需更频繁。启动时仍先清一次。

### 2. 本地镜像删除（prune）：放在镜像同步里

服务器删批次后，它就从 `catalog`（后端 `GET /api/sync/catalog` 下发的"该用户可见的、云端现存批次清单"）消失。因此在 `MirrorSyncAsync` 同步完 catalog 后新增一步 prune：本地有、但 catalog 已无的批次目录直接删除。三条安全设计:

- **只按键 `(流程+组+批次)` 匹配、忽略位置**：凡云端仍存在的批次绝不会被误删，即使本地位置与云端状态暂时不一致也安全。
- **只清 `allowedFlows` 内的流程**：传入操作员 `OperatorIdentity.AllowedFlows`。无任何授权 → 集合为空 → 不动任何本地数据，避免"权限被清空 → 本地被清空"的事故。
- **catalog 拉取失败会先抛异常、根本走不到 prune**：网络抖动不会误删。

顺带效果：超 15 天的旧 Done 孤儿（服务器已删、客户端也早不再同步）现在会被自动清掉。

> 对称语义确认：本地手动删掉一个云端仍存在的批次（Todo 或 15 天内的 Done），下次镜像同步会照 catalog **重建 + 重新下载**把它补回来；本地不是真相源，无法靠删本地永久摆脱云端仍在的批次。数据不丢（行编辑实时写在服务器）。

## 备注 / 待办

- **残余边界**：操作员被**完全撤销所有权限**（`AllowedFlows` 为空）时，prune 不会清其本地历史数据——为安全刻意留白（宁可不删也不误删）。若需"完全撤权也清空本地"，需另加处理。
- **可选增强**：清理间隔（`cleanupInterval`）目前写死 6 小时，可考虑配置化到 `appsettings.json`（如 `Config:CleanupIntervalHours`），以后调频率免改代码重部署。
- **部署**：本次两条线都碰了——后端（清理频率）走后端重部署；客户端 / Infrastructure（prune）需递增 `ApplicationVersion` 重发 ClickOnce。
