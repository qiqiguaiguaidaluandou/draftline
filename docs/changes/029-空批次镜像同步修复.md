# 029 空批次（0 行自动完结）本地镜像同步修复

- 状态：**已实现，待真机验证**
- 日期：2026-07-14
- 关联提交：d7c6607（实现）

> 结论：空批次（取数 0 行、024 引入的"出生即 `Done`"）此前只在**数据窗口 15 天内**才会被客户端镜像下来，
> 导致"服务器上有、本地没有"（尤其追赶旧窗 / 首次部署回填的旧窗空批次）。改为**空批次无条件镜像**，
> 与服务端 Done 保留期对齐，保证"本地 = 云端"。仅一处判据、无迁移、无新表。

## 需求 / 问题（原始现象）

取数时某批次取到 0 行会被直接置「已处理」（见 024）。但操作员反馈：这类空批次**服务器上有、本地却没有**，
期望本地也要有（落在「已处理」页）。

## 现状根因

服务端与客户端对「Done 批次是否近期、要不要留/拉」用了**两套不一致的判据**：

- **服务端保留清理**（`DataIngestionService`）：按 `LastModified`（`RetentionDaysForDone`，默认 15 天）。
  空批次刚落库、`LastModified` 是新的 → **服务端一直留着**。
- **客户端镜像同步**（`BatchSyncService.MirrorSyncAsync`）：对 Done 批次按 `windowEnd`（数据窗口）判 15 天。

而空批次**出生即 `Done`**，客户端从没把它当 `Todo` 同步过——于是只要它的**数据窗口不在 15 天内**
（追赶旧窗 / 首次部署回填等），客户端就永远跳过它，形成"服务器有、本地无"。
（数据窗口本就新的空批次能同步，故此 bug 专坑旧窗空批次。）

## 方案

`MirrorSyncAsync` 的同步范围判定中，**Done 且 `TotalRows == 0` 的空批次无条件纳入同步**，不受 15 天窗口过滤限制：

```csharp
bool isRecentDone   = item.Status == BatchLocation.Done && windowEnd > DateTime.Now.AddDays(-15);
bool isEmptyAutoDone = item.Status == BatchLocation.Done && item.TotalRows == 0; // 新增
bool shouldSync = item.Status == BatchLocation.Todo || isRecentDone || isEmptyAutoDone;
```

关键取舍：

1. **只放开空批次，不动普通批次的近期性语义**（blast radius 最小）。普通有数据的旧 Done 批次仍按 15 天窗口跳过，保持本地精简。
2. **不会无限累积**。空批次体量仅一个空表头 xlsx；且仍受服务端 Done 保留期约束——过期后从 catalog 消失，
   会被 `MirrorSync` 的「镜像删除(prune)」步骤在本地一并清掉，始终"本地 = 云端"。
3. **按 `TotalRows` 判空可靠**：空批次 `TotalRows=0`，正常批次 `>0`；catalog 项的 `TotalRows` 取自 `BatchRegistry`。

## 生命周期（修后每个空批次）

1. 服务端取数 0 行 → 登记 `Done` / `TotalRows=0` / `LastModified=当下` + 落空表头 xlsx。
2. 客户端下次同步（登录 / 手动「同步数据」/ 会话内定时）→ 命中 `isEmptyAutoDone` → 同步到本地「已处理」页（0 行）。
3. 约 15 天后服务端按 `LastModified` 清理 → 从 catalog 消失。
4. 再次同步时「镜像删除」把本地该条也删掉（限有权限的流程内）。

（注：同步在登录 / 手动 / 定时三种时机触发，非取数瞬间即达；只同步操作员有权限的流程/组。）
