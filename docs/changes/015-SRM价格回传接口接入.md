## 需求 / 改动

`docs/接口文档.md` 给出"回传图纸核价价格到 SRM"的接口（点"回传到 SRM"按钮时调用）：

- 请求：`{ "interfaceCode": "AI_ITEM_PRICE_TO_SRM", "content": { "itemList": [ { "itemCode", "price" } ] } }`，`interfaceCode` 固定，`price` 按字符串发。
- 响应：`data[]` 逐条 `{ itemCode, price, remark }`。`remark` 为 `新增成功`/`更新成功` → 该行成功；`失败：…`（如"物料编码在物料主数据中不存在"）→ 该行失败。外层 `code/status` 文档给了两种形态（`"S"` 与 `"200"`+`SUCCESS`），按 `data` 逐行判定即可，不依赖外层。

改前回传全部由 `FakeDataSource`（`ISubmitSink`）顶替。本次把核价→SRM 真实回传接进来。

## 方案

- **接缝不动上层**：`/api/submit` 端点与客户端不改，仍调 `ISubmitSink.SubmitAsync` 拿逐行结果。用 `Srm:Enabled` 开关切换真实实现/Fake（默认 false 走 Fake，不影响本地）。
- **鉴权复用 EBS**：SRM 与 EBS/PLM 用同一套手写 HS256 JWT（同 `Ebs:JwtSecret/JwtIssuer/AuthScheme`），直接复用 `EbsTokenProvider`。
- **只接核价→SRM**：机加挑图→EBS 的回传接口尚未提供，`SrmSubmitSink` 内部对非核价 flow **委托给 FakeDataSource** 占位。
- **字段映射**：`itemCode ← materialCode`（核价行主键）、`price ← targetPrice`（操作员手填，字符串）。
- **逐行成功判定**：响应 `data[].remark` 不以"失败"开头 → 成功；请求行在响应里找不到 → 记为失败（`SRM 未返回该行结果`）。`remark` 原文放进 `SubmitRowResult.Message`。
- **失败语义（用户确认）**：
  - **整批瞬时失败**（HTTP 非 2xx / 响应无 `data` 数组）→ `SubmitAsync` 抛异常 → 端点捕获、记 Failed 日志、批次**不置 Done、可重试**。
  - **逐行永久失败**（如物料不存在，重试也不会成功）→ 转入异常池，与操作员手动挂起的异常同等对待，可在“异常解决”页补回传（待 SRM 主数据修正后重提）。批次照常置 Done。
    - **关键：异常必须写服务端 DB（`db.Exceptions`），不能只写客户端本地。** 异常池是 Remote-First：`BatchSyncService` 会用 `/sync/exceptions` **全量覆盖**本地异常池文件；只写本地（`ILocalBatchStore.AddExceptionsAsync` 写本地 JSON）则下次同步即被冲掉。手动挂起在挂起当刻就调 `/batch/suspend-exception` 落库，故能留存。所以 `/submit` 端点对失败行**直接写 `ExceptionEntity` 到 DB**，与批次置 Done 同一事务，原子且同步必然带回。客户端的本地 `Suspend`+`AddExceptions` 仅作同步前的即时显示。

## 实现

**新增**（`src/Draftline.Gateway/AntiCorruption/`）：
- `SrmOptions.cs`：`Srm` 配置段（`Enabled`、`Url` 占位、`InterfaceCode` 默认 `AI_ITEM_PRICE_TO_SRM`）。
- `SrmSubmitSink.cs`：`ISubmitSink` 真实实现。拼 `itemList` → POST（Bearer JWT）→ 判 HTTP → 解析 `data[]` → 逐行映射回 `RowKey`。非核价 flow 委托 `FakeDataSource`；`ShouldFailBatch()` 恒 false（整批故障靠抛异常表达）。

**改动**：
- `Gateway/Program.cs`：绑定 `SrmOptions`；`Srm:Enabled` 为真时 `AddHttpClient<ISubmitSink, SrmSubmitSink>()`，否则仍用 `FakeDataSource`。
- `Gateway/Endpoints/ApiEndpoints.cs`（`/submit`）：重构失败处理——整批失败（ShouldFailBatch / SubmitAsync 异常）走可重试分支；逐行永久失败**写 `ExceptionEntity` 到云端异常表**（SourceBatch=BatchKey=FolderName，与手动挂起一致）+ 记 Failed 日志，但批次照常完成、把 `RowResults` 回客户端。成功日志 `ImpactCount` 改记实际成功行数。
- `Gateway/appsettings.json`：加 `Srm` 段（占位，`Enabled:false`）。
- `App/ViewModels/BatchWorkViewModel.cs`（客户端）：整批回传后按 `RowResults` 分流——成功行 `MarkUploaded`；被退回的行 `Suspend("…回传失败：remark")` 转入异常池，与手动异常一并 `AddExceptionsAsync`；提示语区分“已回传 / 退回转异常池 / 手动挂起”三类计数。
