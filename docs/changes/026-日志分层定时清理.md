# 026 日志分层定时清理（ActivityLogs 按 Action 分档保留）

- 状态：**已实现，待运行验证**
- 日期：2026-07-13
- 关联提交：5c37559（实现）

> 结论：此前 `ActivityLogs`（全项目唯一一张日志表，靠 `Action` 区分种类）**没有任何清理、无限增长**。
> 现接入定时清理，按性质分三档保留、各自可配。清理挂在 `DataIngestionService` 既有的清理循环里
> （启动清一次 + 每 6 小时一次），与批次清理同节奏。无迁移、无新表。

## 背景

所有日志（回传审计 / 用户操作 / 系统取数 / 后台管理 / 客户端行为埋点）都写进同一张 `ActivityLogs`，
用 `Action` 列区分。此前只有**批次数据文件**有保留期（`RetentionDaysForDone/Todo`），日志表无人清理。

## 方案：按 Action 分三档，各自可配

| 档 | 命中的 Action | 配置键（`Config` 段） | 默认 |
|---|---|---|---|
| 审计 | `Submit`（回传，含 WindowStart/End/AuditId） | `RetentionDaysForAuditLog` | **180 天** |
| 行为埋点 | `Behavior`（客户端 `/oplog` 上报，与各端点动作日志高度冗余） | `RetentionDaysForBehaviorLog` | **30 天** |
| 其余操作/系统 | `Login`/`ChangePassword`/`UpdateRow`/`Suspend`/`Resolve`/`RefetchDrawing`/`Ingest`/`Admin*` | `RetentionDaysForOperationLog` | **90 天** |

- 审计留最久（追溯价值最高，且是回传幂等/补拉判据的依据）。
- 行为埋点最短（信息与 `Submit`/`Resolve` 等端点动作日志重复）。
- 其余归一档 90 天；其中系统 `Ingest` 量最大，如后续要单独更短，再拆一档即可。
- **任一档配 `<= 0` 视为“永久保留”**（跳过该档删除），作为“先不清理”的逃生阀。

## 改动

- `DataIngestionService.cs`：
  - 新增 `CleanupOldLogsAsync`：三档各用 `ExecuteDeleteAsync` 直接下发 `DELETE`（不把行读进内存）。
  - 在启动清理、周期清理（`cleanupInterval=6h`）两处，紧跟批次清理各调一次。
- `appsettings.json` `Config` 段：加 `RetentionDaysForAuditLog:180` / `RetentionDaysForOperationLog:90`
  / `RetentionDaysForBehaviorLog:30`（真实值可在 gitignored 的 `.local.json` 覆盖；不配则用代码默认）。

## 待验证

- 运行态确认：跑起来后日志出现 `Log cleanup completed. Removed N ...`，且各档只删到各自阈值以外的行。
- `ExecuteDeleteAsync` 走 Npgsql（EF 8）单条 DELETE，无需迁移；生产库首次清理若历史积压多，DELETE 量可能较大，
  但按 Action+Timestamp 有索引（`idx_activity_log_user_time` 等），可接受。
