# 032 修复挑图本地过期批次不被镜像删除（prune）

- 状态：**实现完成，待运行态验证**
- 日期：2026-07-16
- 关联提交：`01abcfa`（fix(sync): 修复挑图本地过期批次不被镜像删除（032））

## 需求 / 改动

测试定期清理时把 `Config:RetentionDaysForDone` 由 15 改为 2 天并重启后端，观察到：

- 服务器磁盘 + 数据库里的过期「已处理」数据都被清掉了（服务端清理正常）；
- 但**客户端本地**：核价的过期批次文件夹删掉了，**挑图的却没删**。

预期两个流程一致——云端过期清理后，本地对应批次也应随同步一并消失。

## 根因

本地文件的删除并非服务端清理所为，而是客户端同步时的「镜像删除（prune）」：`BatchSyncService.MirrorSyncAsync` 第 2 步会删掉"本地有、但云端 catalog 已无"的批次目录。**这一步对挑图失效。**

问题出在**本地目录层级两个流程不对称**（`LocalPaths.LocalGroupRoot`）：

```
核价(Pricing):        位置/<组名>/<批次>      ← 有"组"这一层
挑图(DrawingSelection): 位置/<批次>            ← 无"组"层（组名恒为 "Center"，只存在于 DB/catalog，本地路径与页面都不体现）
```

而 prune 的目录枚举 `EnumerateLocalBatchDirs` **硬编码了核价的两层结构**：把状态目录下第一层当"组"、再向下找"批次"。对挑图而言第一层其实就是批次目录、其下没有子目录，于是**挑图批次一个都枚举不到 → 永远不进入 prune 判定 → 永不删除**；核价因层级吻合而正常。

一处佐证不一致：读取侧 `LocalBatchStore`（`:53`）读本地挑图批次时，早已按"扁平目录 + 组名硬填 `Center`"正确处理；偏偏 prune 的枚举没按同一规则来——同一个"挑图目录长什么样"，读取侧对、prune 侧错，即为此 bug。

## 方案

**最小改动，只修 prune 的目录枚举**，不动其它任何逻辑：

- `EnumerateLocalBatchDirs` 增加 `flow` 参数，按流程感知层级：
  - 挑图：状态目录下第一层直接当批次目录，组名固定 `"Center"`——必须与云端 catalog 里挑图 `GroupName="Center"` 对齐，否则镜像删除的键（流程+组+批次）匹配不上；
  - 核价：保持原两层枚举，**逐字不变**。
- 唯一调用点（prune 循环，已有 `flow` 变量）把 `flow` 传入。
- 更正该方法上过时/误导的注释（原写"取两层"）。

**未做**（按"最小改动"约定，留待后续治本）：消除本地目录层级的流程不对称、把散落的 `"Center"` 字面量收敛为常量。

## 实现

- `src/Draftline.Infrastructure/Sync/BatchSyncService.cs`：
  - `EnumerateLocalBatchDirs(string locationRoot, FlowType flow)` 加 `flow` 参数 + 挑图扁平枚举分支（组名 `"Center"`）；核价分支原样保留；更正注释。
  - prune 调用点 `EnumerateLocalBatchDirs(locationRoot, flow)`。
- `tests/Draftline.Tests/Sync/BatchSyncServiceTests.cs`：新增 `MirrorSync_prunes_local_drawing_batch_absent_from_catalog`——本地一个挑图 Done 批次云端已无 → 应删；另一个云端仍在 → 应留。

## 验证

- `dotnet test tests/Draftline.Tests -p:EnableWindowsTargeting=true`：**105/105 全过**（含新增挑图 prune 用例；原核价 prune 用例仍过）。
- 影响面：`EnumerateLocalBatchDirs` 全局仅 prune 一处调用；核价分支字节级未改、行为不变；仅挑图 prune 由"漏删"变"按 catalog 正确删除"。**未触碰**：批次同步下载、状态位置移动、异常池同步、服务端定期清理。
- 运行态待真机确认：把挑图某已处理批次在云端过期清理后，客户端登录/定时同步一次，本地对应挑图文件夹应随之删除，且云端仍在的挑图批次不被误删。
