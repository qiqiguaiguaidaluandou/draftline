## 需求 / 改动

对 Remote-First 后端做一轮**不依赖外部接口**的正确性与安全加固。这些问题在"等对接"阶段不影响演示，但真接口接上后会直接造成数据损坏、对账事故或越权，故先于对接修掉。逐项现状：

1. **服务端改 Excel 无并发锁**（`Stores/FileServerBatchStore.cs:92-155`）
   `UpdateExcelRowAsync` 是"读整本 → 改 → `FileMode.Create+FileShare.None` 写回"。客户端自动保存为 800ms 去抖高频调用，多端/多字段并发改同一批次会 `IOException` 或写坏 xlsx。文档 ⑦ §5 明确要求"并发锁"，但代码未实现。

2. **`/submit` 非事务、无幂等、不处理失败**（`Endpoints/ApiEndpoints.cs:254-292`）
   - `registry.Status = Done` 与 `ActivityLog.Status="Success"` 无条件写死（`:263`、`:277`），即使回传结果含失败行也照样成功。
   - `ShouldFailBatch()` 为真时（`:257`）直接返回 `Success=false`，但**不记任何 Failed 日志、不改状态**，失败无痕。
   - 无幂等：重复提交会重复回传 + 重复置 Done。

3. **补拉判据靠字符串匹配，脆弱**（`/audit/exists`，`Endpoints/ApiEndpoints.cs:316-332`）
   用 `Payload.Contains(windowStartString)` 判断窗口是否提交过、用 `Split("AuditId: ")` 反解析 AuditId，且不校验 windowEnd。`Payload` 文本格式一改即失效。

4. **登录自动放权，白名单被架空**（`Endpoints/ApiEndpoints.cs:28-35`）
   任何工号首次登录自动 `Add` 组1 两流程权限，使 doc ⑧ 的 "Deny-All 白名单" 形同虚设。

5. **`resolve-exception` 授权未校验 Flow**（`Endpoints/ApiEndpoints.cs:220-221`）
   只校验 `GroupName`，未校验 `Flow`，存在跨流程越权解决异常的理论口子（其余写端点均已校验 flow）。

6. **死代码**：`Endpoints/ApiEndpoints.SaveGroupBatchAsync`（`:335`）无任何调用（采集逻辑在 `DataIngestionService`），属遗留。

## 方案

一次「后端加固」变更覆盖以上 6 项，原则：**只动后端、不改客户端 UI/契约形状、可在 Linux 编译+测试验证**。

### 1. Excel 并发锁
在 `FileServerBatchStore` 内引入按批次键串行化的写锁：
- 用 `ConcurrentDictionary<string, SemaphoreSlim>`，键 = `$"{flow}/{group}/{batchId}"`。
- `UpdateExcelRowAsync`（写）与 `SaveBatchAsync`（建批次写整本）进入前 `await sem.WaitAsync(ct)`，`finally` 释放。
- `OpenFile`/`ListFilesAsync`（只读）不加锁（读不会损坏，且 catalog 高频）。
- 取舍：进程内锁，够用（单网关实例部署，见 `方案设计.md §10`）；多实例部署需改分布式锁，留待 备注。

### 2. submit 成功判定 + 幂等 + 失败留痕
重写 `/submit` 处理逻辑（语义按 `方案设计.md §7/§9`）：
- **幂等**：先查 `registry`，若 `registry.Status == Done` 则视为已提交，**不再调 `sink`**，直接返回 `Success=true` + 既有 AuditId（取自 registry，见第 3 点新增列），消息标注"已提交过"。
- **失败留痕**：`ShouldFailBatch()` 为真，或 `rowResults` 含 `Success=false`，均视为失败——记一条 `ActivityLog{Action="Submit", Status="Failed", Payload=失败摘要}`，`registry` **不**置 Done，返回 `Success=false` + `RowResults`（供客户端按行重试）。
- **成功**：仅当 `!ShouldFailBatch()` 且所有行成功，才置 `registry.Status=Done`、写结构化审计、回传 AuditId。
- 暂不实现"服务端读自身磁盘 Excel 再回传"（doc ⑦ 设想）：当前 `FakeSink` 不校验行内容，留待真 `ISubmitSink` 对接时一并做，备注记录。

### 3. 审计结构化（顺带支撑第 2 点幂等）
给 `ActivityLog` 增列 `WindowStart`/`WindowEnd`（`DateTime?`）、`AuditId`（`string?`）；给 `BatchRegistry` 增列 `AuditId`（`string?`，提交成功时写入，支撑幂等回显）。
- `/submit` 成功时把 `WindowStart/WindowEnd/AuditId` 写进结构化列（不再只塞 `Payload`）。
- `/audit/exists` 改为按 `Flow + EmployeeId + Action="Submit" + WindowStart==ws + WindowEnd==we` 精确查，直接读 `AuditId` 列，删除字符串反解析。
- 生成 EF 迁移 `HardenSubmitAndAudit`（新增可空列，向后兼容；旧行结构化列为 null 不影响）。

### 4. 登录自动放权加环境围栏
`/api/auth/login` 处理器注入 `IHostEnvironment env`，把 Auto-Seed 权限包进 `if (env.IsDevelopment())`。生产环境不自动放权（白名单生效）。
- 取舍：生产暂无 ACL 管理入口，需 DBA 直接写 `UserPermissions` 或待 DHR/管理后台对接；记入 备注。

### 5. resolve-exception 补 Flow 校验
`SuspendExceptionRequest` 已带 `Flow`，但 `resolve-exception` 当前签名只收 `groupName/batchId/rowKey`。增加 `flow` 查询参数，权限校验改为 `p.Flow == flow && (p.GroupName=="*" || p.GroupName==groupName)`，并在 `Exceptions` 查询时带上 `Flow`。
- 客户端 `HttpDataGateway.ResolveExceptionAsync` 与 `IDataGateway` 契约同步加 `FlowType flow` 参数；调用方（`BatchWorkViewModel`/`ExceptionResolveViewModel`）传入当前 flow。这是契约小幅扩展，不改 UI 表现。

### 6. 删除死代码
移除 `ApiEndpoints.SaveGroupBatchAsync` 及其未使用的 using（如有）。

## 实现

改动文件：

**1. Excel 并发锁** — `Stores/FileServerBatchStore.cs`
- 新增静态 `ConcurrentDictionary<string, SemaphoreSlim> _writeLocks` + `LockFor(flow,group,batchId)`。
- `SaveBatchAsync`、`UpdateExcelRowAsync` 写入全程 `await sem.WaitAsync(ct)` / `finally Release()`；只读方法不加锁。

**2. submit 重写** — `Endpoints/ApiEndpoints.cs` `/submit`
- 先查 registry；`Status==Done` → 幂等返回既有 `registry.AuditId`，不再调 sink。
- `ShouldFailBatch()` 或任一行 `Success==false` → 记 `ActivityLog{Status="Failed", Payload=失败行}`，不置 Done，返回 `Success=false`+`RowResults`。
- 全部成功才置 `Status=Done`、写 `registry.AuditId` 与结构化审计（`WindowStart/WindowEnd/AuditId` 独立列）。

**3. 审计结构化** — `Stores/ActivityLog.cs`（+`WindowStart/WindowEnd/AuditId`）、`Stores/BatchRegistry.cs`（+`AuditId`）、`Stores/DraftlineDbContext.cs`（+索引 `idx_activity_log_audit_lookup`）、`/audit/exists` 改为结构化精确查并直读 `AuditId` 列。
- 迁移：手写 `Migrations/20260618000000_HardenSubmitAndAudit.cs`(+`.Designer.cs`) 并同步 `DraftlineDbContextModelSnapshot.cs`（4 个可空列 + 1 索引；本环境无 SDK，未用 `dotnet ef` 生成，结构对照上一迁移人工编写）。

**4. 登录围栏** — `/api/auth/login` 注入 `IHostEnvironment`，Auto-Seed 包进 `if (env.IsDevelopment())`。

**5. resolve-exception 补 Flow** — `/batch/resolve-exception` 增 `flow` 参数 + 权限/查询带 Flow；契约 `IDataGateway.ResolveExceptionAsync` 与 `HttpDataGateway` 加 `FlowType flow`；调用方 `BatchWorkViewModel`、`ExceptionResolveViewModel` 与测试 `FakeDataGateway` 同步。

**6. 删除死代码** — 移除 `ApiEndpoints.SaveGroupBatchAsync`。

**测试** — 新增 `tests/Draftline.Tests/Storage/ServerBatchStoreConcurrencyTests.cs`：并发 `UpdateExcelRowAsync` 不抛错且各行值正确。
