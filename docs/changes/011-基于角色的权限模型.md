# 011 基于角色的权限模型（Role）

- 状态：已完成（编译/测试/迁移一致性均已验证；尚未提交 git、WPF 真机待验）
- 日期：2026-06-22
- 关联提交：<git 短哈希>（提交时填）

## 需求 / 改动

权限的唯一目的是**区分每个用户能查看哪些数据**——能看到某批次/表单即可操作它，不区分"只读/可编辑/可提交"等功能级权限。改前权限是逐用户直配 `UserPermission(工号, 流程, 组)`，多用户时一条条配、繁琐易错，且没有"角色"这种可复用的范围捆绑。

本次引入**角色（Role）机制**：定义形如"核价员-组1"的角色（打包若干 `流程+组` 数据范围），给用户挂角色，自动展开成该用户的有效可见范围。纯角色制——不再保留逐用户直配。

附带修掉的问题：
- 6 个数据端点各自**内联**了几乎相同的权限判断（`UserPermissions.Where/AnyAsync(... * or ==group ...)`），重复且易漂移。
- `/submit` 端点此前**完全没有**数据范围校验（仅按 batchKey 找批次），构造请求可越权回传——补上校验。

## 方案

- 三张表替代 `UserPermission`：
  - `Role(Id, Name 唯一, Description)`：角色 = 命名的数据范围捆绑。
  - `RolePermission(RoleId, Flow, GroupName)`：角色含的一条范围（`*`=该流程全部组）；`(RoleId,Flow,Group)` 唯一；删角色级联删。
  - `UserRole(EmployeeId, RoleId)`：用户↔角色多对多；`(工号,RoleId)` 唯一；删角色级联删指派。
- **有效可见范围 = 用户所有角色的 (流程,组) 并集**，语义与原 `UserPermission` 完全一致，故端点的"圈数据"逻辑不变，只是来源换成角色展开。
- 新增 `IPermissionService`（`GetGrantsAsync`/`GetFlowsAsync`/`HasAccessAsync`）作为"能看哪些数据"的唯一入口，6 个数据端点与 `DbAuthService.AllowedFlows` 全部改走它，消除重复内联。
- 管理端（仍 REST，仅启用中管理员）：
  - 角色：`GET/POST /admin/roles`、`PUT/DELETE /admin/roles/{id}`（范围覆盖式设置）。
  - 给用户挂角色：`PUT /admin/users/{工号}/roles`（覆盖式，校验角色存在）。
  - 移除 `PUT /admin/users/{工号}/permissions`；`GET /admin/users` 改为返回所挂角色 + 展开后的有效范围。
- 取舍：纯角色制（删直配）。目前尚无真实权限数据（认证此前是占位），迁移代价小、模型最干净，契合"权限只为圈数据"的诉求。改角色即时对所有挂载用户生效，无需逐人重配。

## 实现

改动文件：

**Server** — 删 `Stores/UserPermission.cs`；新增 `Stores/Role.cs`、`Stores/RolePermission.cs`、`Stores/UserRole.cs`、`Auth/PermissionService.cs`（含 `Grant` record struct + `IPermissionService`）；`Stores/DraftlineDbContext.cs`（换 DbSet、加 3 表的唯一索引与 FK 级联）；`Auth/IAuthService.cs`（`DbAuthService` 注入 `IPermissionService`，`AllowedFlows` 走它）；`Endpoints/ApiEndpoints.cs`（6 个数据端点改用 `IPermissionService`；`/submit` 补范围校验；重写 `MapAdminApi`：角色 CRUD + 挂角色，删直配，用户列表带角色/有效范围；新增 `BuildRolePermissions` 辅助）；`Program.cs`（注册 `IPermissionService`）。

**Core 契约** — `Contracts/Http/HttpDtos.cs`：删 `SetPermissionsRequest`；新增 `RoleRef`/`RoleSummary`/`SaveRoleRequest`/`SetUserRolesRequest`；`UserSummary` 由 `Permissions` 改为 `Roles` + `EffectivePermissions`；`PermissionDto` 保留（角色定义与有效范围展示共用）。

**迁移** — `dotnet ef migrations add AddRoles`：`DropTable UserPermissions` + 建 `Roles`/`RolePermissions`/`UserRoles`（FK 级联 + 唯一索引）；Down 可逆。带数据丢失提示（即删 `UserPermissions`，符合预期）。

**测试** — 新增 `tests/Draftline.Tests/Auth/PermissionServiceTests.cs`（无角色→空、单角色展开、多角色并集、`*` 通配、重叠去重）；改 `DbAuthServiceTests`（`NewService` 注入 `PermissionService`，权限来源由 `UserPermission` 改为角色；新增 `SeedRole` 辅助）。

**客户端** — 不动：WPF 只消费服务端返回的"可见数据"，权限模型变化对其透明。

## 备注 / 待办

- 现有部署若已有 `UserPermissions` 数据，本迁移会**删表丢数据**（本项目此前为占位认证、无真实权限数据，影响可忽略）；如需保留请在 `database update` 前自行备份/手工迁移到角色。
- 角色管理目前仅 REST，无 WPF 管理界面（单独立项）。
- 组名仍是自由字符串（配错只是拿不到数据、不报错）；可加"可选组列表"端点辅助管理端做下拉/校验——单独立项。
- 部署操作（配 `Jwt:Key`/`Admin`、`database update`、建角色挂角色）见 010 备注与部署说明。
