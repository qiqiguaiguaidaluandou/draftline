# 023 取数时间窗变更后的取数正确性（防倒补 / 防断档）

- 状态：**已实现（D 方法：数据时间轴高水位 `T_covered`），待真机验证**
- 日期：2026-07-08（定稿+实现 2026-07-09）
- 关联提交：<待提交时补>

> 结论：采用 **D 方法**——每流程维护"数据时间轴高水位 `T_covered`"，取数从上次采到处向前接续。
> `T_covered` 从现有 `BatchRegistries` **派生**（不新增存储），配合 `effStart = max(T_covered+1, ws)`
> 与"全 horizon 枚举"，使日常/宕机补采**都产出标准批次**，同时改窗不倒补、不丢数。
> 「讨论进展」保留 A/B/C/D 的推演脉络作为背景；「定稿方案」为最终设计。

## 需求 / 改动（原始问题）

`DataIngestionService.CheckSchedulesAsync` **每 5 分钟一轮、且每轮都**回看「昨天 + 今天」两天的触发时刻，
对 `now >= triggerAt` 且对应批次未登记的窗口去 EBS 取数。批次键 `batchId = BatchFolderName(窗口起, 窗口止)`
由时间窗解算而来（`IngestionSchedules` 里 `StartDayOffset/StartTime/EndTime` 决定）。

**问题**：改了采集时间窗并重新部署重启后，窗口变 → `batchId` 变 → 回看昨天/今天已过去的触发点时，
新 `batchId` 未登记 → 会**用新窗口倒着把过去最多两天的批次补出来**。期望：改窗后**不倒补过去**，只随时间
到达新的取数节点再取。

> 采集排程当前**硬编码**在 `src/Draftline.Core/Schemas/IngestionSchedules.cs`，改窗 = 改代码 + 重新部署重启。

## 现状基线：窗口是"无缝平铺"的

窗口经 `CollectionWindow.Resolve(anchor)` 解算：`起 = (anchor+StartDayOffset) 的 StartTime`，`止 = anchor 的 EndTime`。
当前两套排程严格首尾相接、绕 24h 闭合:

- **核价（2 窗/天）**：`[D-1 15:31→D 09:30]`（触发10:00）、`[D 09:31→D 15:30]`（触发16:00）。
- **挑图（3 窗/天）**：`[D-1 18:01→D 09:30]`（10:00）、`[D 09:31→D 14:30]`（15:00）、`[D 14:31→D 18:00]`（18:30）。

**不变式**：`本窗.起 = 相邻前窗.止 + 1 分钟`，且绕 24h 闭合。满足它 → 时间轴不重不漏。

## 讨论进展（推演脉络，按时间顺序，作为背景保留）

### A. 最初方案：排程指纹 + `EffectiveFrom`（按触发时刻 forward-only）

- 持久化每流程的排程指纹；启动时比对：无记录→存指纹、`EffectiveFrom=远古`（首次维持现状）；一致→不变
  （宕机补采照旧）；不一致（改了窗）→ `EffectiveFrom=now`、更新指纹。
- `CheckSchedulesAsync` 每轮对每个触发点判 `triggerAt < EffectiveFrom → 跳过`。
- 已澄清：过滤放在**轮询内**（每 5 分钟都判），故轮询不会绕过边界去倒补；`EffectiveFrom` 只在启动**更新其值**，
  但**读取+比较每轮都做**。

### B. 发现风险 1：改坏平铺 → 每天复发的静默断档（与 EffectiveFrom 无关）

只改一个窗、不同步改相邻窗，`起=前窗.止+1min` 被破坏:
- **例**：把核价「上午批」起点 `15:31→16:00`，但「下午批」仍止于 `15:30` → `[15:31, 15:59]` 无窗覆盖。
- 这不是一次性的：两窗每天在此衔接 → **每天都丢这半小时**，EBS 该时段物料永远取不到，且**不报错**。
- 根因：**只挪了一条边**。正确做法是"成对挪边界"（要挪起点，就把相邻窗的止点也挪到 起点-1min）。

### C. 发现风险 2：即使平铺改对，**切换当天仍会漏一次**（EffectiveFrom 的固有缺陷）

按正确改法（同时把「下午批」止改到 `15:59`、「上午批」起改到 `16:00`），D 日 20:00 部署:
- 旧「下午批」（止15:30，触发 D 16:00）**已跑**，只取到 15:30；
- 新「下午批」（止15:59，触发 D 16:00）在部署前 → `triggerAt < EffectiveFrom` → **被跳过**；
- 新「上午批」（起16:00，触发 D+1 10:00）从 16:00 才起。
- → `[D 15:31, D 15:59]` 三不管，**这一天丢一次**。根因：这段的"责任触发点"（D 16:00）在部署前已按
  旧定义跑过、按新定义又被 forward-only 跳过。**只要挪动了边界，切换当天必漏"挪动的那一段"。**

> 结论：**指纹 + EffectiveFrom（按触发时刻）不足以保证不丢数**——它管不住风险 2，也管不住风险 1。故弃用 A。

### D. 选定方案：按「数据时间轴」记覆盖高水位 `T_covered`

每个流程记 `T_covered = 已连续采到的数据时间`，取数从上次采到处向前接续。经过几轮细化（派生存储、
`max(T_covered+1, ws)`、全 horizon 枚举、分组失败两层防护），确定为**定稿方案**，详见下节。

### E. 关于"闭环改窗规范"能否简化方法（澄清）

若操作上保证"闭环改窗"（成对挪边界、平铺不破坏），只能消掉**风险 1**；**倒补与风险 2 与改窗是否闭环无关**，
仍需 D 方法治理。故闭环规范是 D 的**互补流程规范**，不能替代 D。好处：平铺自检可降级为可选（甚至不做）。

### F. 关于"宕机补采产出拉长追赶窗"的修正（重要）

早期草案让"最早候选窗口吞下全部积压"，会产出跨多天的非标准"追赶窗"。**已修正**：改为
①回看范围 = 追赶上限、枚举每一个错过的**标准**窗口；②`effStart = max(T_covered+1, ws)`。
因平铺无缝，日常与宕机补采下 `T_covered+1 == 下一窗标准 ws`，故**每个补采批次都是标准窗口**；
仅当"改窗把窗口止往后延、且部署时该窗触发点已过"时，才产生**一次性**的非标准过渡批次（不可避免，见「取舍」）。

## 定稿方案（D 方法）

### 1. 核心概念

每个流程维护**数据时间轴高水位 `T_covered`**：这条流程的数据**已连续采到哪个时刻**。
取数不再死取排程窗口，而是**从 `T_covered` 之后接续采到本次窗口止**。

### 2. `T_covered` 从哪来：从 `BatchRegistries` 派生，不新增任何存储

不建表、不加迁移、不写 JSON。已落盘登记的批次文件夹名（`batchId`）里就编码了窗口止
（`LocalPaths.BatchFolderName` ↔ `TryParseFolderName`）。故：

```
T_covered(flow) = max( 解析出的 we )
                  取自该 flow 下「期望组已登记齐全」的所有 batchId
无任何合格批次 → null（首次部署 / 长期闲置）
```

要点：
- `we` 是完整 DateTime（含日期），跨天单调递增，`max` 即"最近采完的窗口止"，无 24h 绕回问题。
- **"期望组齐全才算数"**：核价须组1、组2 都登记了该 batchId 才计入 `max`；只登记一组的（部分失败）
  **不计入** → `T_covered` 不会越过它 → 留给组内自愈补齐（见 §5）。
- `expectedGroups`：核价 = `_ebs.PricingGroups`，挑图 = `["Center"]`——与 `IngestWindowAsync` 内一致，必须共用同源。

派生函数（伪码，命名对齐现有代码）：

```csharp
private async Task<DateTime?> ComputeTCoveredAsync(
    DraftlineDbContext db, FlowType flow, string[] expectedGroups, CancellationToken ct)
{
    var rows = await db.BatchRegistries
        .Where(b => b.Flow == flow)
        .Select(b => new { b.BatchId, b.GroupName })
        .ToListAsync(ct);

    DateTime? tc = null;
    foreach (var g in rows.GroupBy(x => x.BatchId))
    {
        var groups = g.Select(x => x.GroupName).ToHashSet();
        if (!expectedGroups.All(groups.Contains)) continue;          // 组不齐 → 不算采完
        if (!LocalPaths.TryParseFolderName(g.Key, out _, out var we)) continue;
        if (tc is null || we > tc) tc = we;
    }
    return tc;
}
```

### 3. 判断"该不该取"

两个条件同时满足才取，缺一不可：
1. **到点了**：`now >= triggerAt`——比窗口止晚约 30 分钟等数据落定，不取未封口窗口。（**沿用现状**）
2. **还没采到**：`窗口止 > T_covered`——数据时间轴尚未覆盖该窗口。（**新增闸门**）

再加一层：进了窗口后，现有"逐组预检"仍判该 batchId 是否已登记+落盘，已齐即跳过，防重复调 EBS。

### 4. 取数起点与追赶：`effStart = max(T_covered+1, ws)` + 全 horizon 枚举

`CheckSchedulesAsync` 核心循环（伪码）：

```csharp
// 回看的日历天数（含今天）。默认 2 = 昨天+今天，对齐现状 `new[]{ now.Date.AddDays(-1), now.Date }`。
var horizonDays = _config.GetValue<int>("Ingestion:MaxCatchupDays", 2);

// 每流程：先算 T_covered
var expectedGroups = flow == FlowType.Pricing ? _ebs.PricingGroups : new[] { "Center" };
var tCovered = await ComputeTCoveredAsync(db, flow, expectedGroups, ct);

// 回看 horizon 个日历天（含今天），枚举每一个已到触发时刻的【标准】窗口
var candidates = new List<(DateTime ws, DateTime we)>();
for (int d = horizonDays - 1; d >= 0; d--)   // horizonDays=2 → d=1,0 → 昨天、今天
{
    var triggerDate = now.Date.AddDays(-d);
    var anchor = DateOnly.FromDateTime(triggerDate);
    foreach (var s in IngestionSchedules.For(flow).Where(s => s.Enabled))
    {
        if (now < triggerDate.Add(s.TriggerTime.ToTimeSpan())) continue; // 没到点不采
        var (ws, we) = s.ToWindow().Resolve(anchor);
        candidates.Add((ws, we));
    }
}
candidates.Sort((a, b) => a.we.CompareTo(b.we));   // 按窗口止升序，逐窗推进

foreach (var (ws, we) in candidates)
{
    if (ct.IsCancellationRequested) return;
    if (tCovered is DateTime tc && we <= tc) continue;           // 已覆盖 → 跳过（不倒补）
    var effStart = tCovered is DateTime t && t.AddMinutes(1) > ws
                   ? t.AddMinutes(1)      // 与已采数据重叠(改窗延长) → 从接续点起，不倒补
                   : ws;                  // 正常/宕机 → 标准窗口起点
    try
    {
        await IngestWindowAsync(db, source, store, flow, effStart, we, ct);
        tCovered = we;                    // 取成功才推进（供本轮后续候选接续）
    }
    catch (Exception ex)
    {
        _logger.LogError(ex, "采集失败：{Flow} 窗口 {S:yyyy-MM-dd HH:mm}~{E:yyyy-MM-dd HH:mm}", flow, effStart, we);
        break;                            // 本流程本轮停在失败窗口，下轮重试，避免越过"洞"
    }
}
```

**为什么日常/宕机补采都是标准批次**：排程无缝平铺 ⇒ `T_covered+1 == 下一窗标准 ws`，故 `effStart == ws`，
每个批次都是标准 `[ws, we]`。`max` 仅在"窗口与已采数据重叠"（改窗把窗口止往后延）时取 `T_covered+1`，
此时取标准窗口必然倒补，故取接续点是唯一正确选择。

**追赶上限**：回看范围即 horizon（`Ingestion:MaxCatchupDays`，**默认 2 = 昨天+今天，对齐现状**）。超出 horizon 的
更久远数据不补，因 horizon 外的窗口根本不进入候选、其数据被跳过——如需显式告警可在最早候选处比对 `T_covered`
落后程度并 `LogWarning`。宕机若超过 2 天，可临时调大本值再重启以补更久的历史。

### 5. 分组部分失败：两层防护（不丢组数据）

- `IngestWindowAsync` 逐组落盘，某组抛异常 → 整调用 throw → §4 catch 到 → **不执行 `tCovered=we`** 且 `break`；
- 该 batchId 只有部分组登记 → 下轮 `ComputeTCovered` 判"期望组不齐" → **不计入 max** → `T_covered` 仍在此窗前；
- 下轮该窗 `we>T_covered` → 重进 `IngestWindowAsync` → 逐组预检发现缺组 → 只补缺组。**不丢数。**

即：`T_covered` 是"要不要考虑此窗"的**粗闸门**；窗内是否重取、补哪些组，仍由现有**逐组预检**（保留不动）决定。

### 6. 运行期持续自检（回答"运行中还查不查漏采"）

`ExecuteAsync` 的 5 分钟轮询**只要后端在跑就一直转**，每轮都跑 `CheckSchedulesAsync`。而
"`窗口止 > T_covered` 就取"这一步本身**就是**"这个窗口采到没有"的检查——**内建在每轮轮询里，不是启动一次性动作**。
无需单独的"扫漏"模块：`T_covered` 是高水位，任何 `窗口止 > T_covered` 且在 horizon 内的窗口按定义即"漏采"，
自动被捞出重取。故**瞬时故障自愈**：某窗触发时 EBS 抽风取数失败 → 未登记 → `T_covered` 不前进 →
下一轮（仍在 horizon 内）自动重试，直到成功。

**边界（如实说明）**：D **只往前看**（`窗口止 > T_covered`），**不回翻高水位以下已覆盖的窗口**。正常运行下
"平铺无缝 + 失败即 break + 期望组齐全才推进"三条保证高水位以下不会凭空出现洞；清理服务删的旧批次是有意删的，
不该重取。唯一 D 不自愈的：**外力**（手动删中间近期批次 / 存储损坏）在高水位以下留洞——属运维范畴，
可临时用 `Ingestion:NowOverride` 或人工干预重取，非常态。

### 7. 不可回避的取舍（已选 D）

**"批次永远标准" 与 "改窗延长时不丢被挪出的碎片" 不可能同时成立**——被挪出的碎片（如 `[15:31,15:59]`）
本身不是标准窗口。二选一：
- **D（已选）**：改窗延长的那一批非标准，但**不丢数、不倒补**；且能靠"改窗提前部署"规范把它也消掉。
- A（纯标准窗口）：批次永远标准，但改窗延长当天丢碎片一次（除非提前部署）。已因风险 2 弃用。

**配套规范**：改窗时**赶在当天受影响的触发点之前部署**（如早上 10:00 前），新窗口从当天首个触发点即按新标准生效，
连那一个非标准过渡批次都不会出现。即使某次没守规范（被迫下午部署），D 也只是多一个非标准批次而**不丢数**。

### 8. 与闭环改窗规范的分工

| 职责 | 由谁保证 |
|---|---|
| 不倒补、切换当天不漏（风险 2） | **代码**：`T_covered` |
| 平铺不破坏、不每天丢半小时（风险 1） | **流程**：闭环改窗规范（成对挪边界） |
| 组内部分失败自愈 | **代码**：现有逐组预检（保留） |

因风险 1 交给流程纪律，**"排程平铺自检"启动校验可不做**——D 的自动缝合能力仅作安全网。

### 9. 新增配置

```jsonc
// appsettings(.local).json
"Ingestion": {
  "MaxCatchupDays": 2   // 回看/追赶上限（日历天，含今天）；默认 2 = 昨天+今天，对齐现状。日常无关，仅决定宕机后向前补多少天；宕机久了可临时调大。
}
```
（`Ingestion:NowOverride` 保持现状不动。）

## 实现（已落地，blast radius 很小）

- **新增** `src/Draftline.Core/Schemas/IngestionPlanner.cs`（纯逻辑、无副作用、可单测）：
  - `HighWatermark(registered, expectedGroups)`：从已登记批次派生 `T_covered`（期望组齐全才计入）；
  - `Plan(now, tCovered, horizonDays, schedules)`：全 horizon 枚举 + `we>T_covered` 闸门 + `effStart=max(T_covered+1, ws)`，
    产出按窗口止升序的 `PlannedWindow` 序列（happy-path 递推推进 `T_covered`）。
- `src/Draftline.Gateway/DataIngestionService.cs`：
  - 新增 `ComputeTCoveredAsync`（查 `BatchRegistries` → 委托 `HighWatermark`）；
  - 改写 `CheckSchedulesAsync`（逐流程算 `T_covered` → `Plan` → 逐窗取数，失败即 `break`）；
  - 加"追赶超上限"`LogWarning`（`plan[0].EffStart > T_covered+1min` 时留痕丢弃的历史段）；
  - `IngestWindowAsync` **签名与内部不变**，仅由调用方传入 `effStart`（原为窗口起）。
- `src/Draftline.Gateway/appsettings.json`：`Ingestion` 段加 `"MaxCatchupDays": 2`。
- **新增测试** `tests/Draftline.Tests/Schemas/IngestionPlannerTests.cs`（10 例，覆盖下节验证方向）。
- **无数据库迁移、无新模型、无新表。**
- 待办：在改窗操作规范处强调"成对挪边界、优先在受影响触发点之前部署"（见备注）。

## 验证

已落地单测 `IngestionPlannerTests`（10 例，全绿），覆盖纯决策逻辑：

1. **稳态等价** ✅：连续两天正常触发，产出窗口与排程逐字一致（`effStart==ws`）。
2. **不倒补** ✅：`T_covered` 在最新时 plan 为空；`we≤T_covered` 的过去窗口一律跳过。
3. **风险 2 不漏** ✅：改窗延长（下午批止 15:30→15:59）D 日 20:00 部署，断言只取 `[15:31,15:59]` 且未重取 `[09:31,15:30]`。
4. **宕机补采出标准批次** ✅：`T_covered` 落后 2 天、horizon=2，断言补出逐个**标准**窗口、**无**跨天拉长窗。
5. **追赶上限** ✅：`T_covered` 落后 5 天、horizon=2，断言只补 2 天、更久远不补（最早窗口仍标准起点）。
6. **分组部分失败** ✅：`HighWatermark` 对只登记部分组的 batchId 不计入 → `T_covered` 不越过（组内自愈由现有逐组预检承接）。
7. **首次部署 / 闲置退化** ✅：`T_covered=null` → `effStart=ws`（等同现状"用窗口起"）。
8. **未到点不采** ✅：触发时刻未到的窗口不进入 plan。

**待真机验证**（依赖 EBS/DB，未在单测覆盖）：整链取数落盘、运行期瞬时故障跨轮自愈、分组部分失败后次轮只补缺组。
