# 019 日志可读性改进 + 操作员/后台日志分流 + 采集计划归位

- 状态：已完成
- 日期：2026-07-01
- 关联提交：6b22ebe、bb1c468、01caa9e

## 需求 / 改动

三组问题，围绕「日志读得懂」与「采集计划该放哪」：

1. **日志中文化**：后台操作日志的动作（`action`）是英文（`Login/Submit/…`），不够清晰；「影响」列含义不明；「详情」列内容（英文短句 + 技术 payload）看不懂。
2. **提交回传太笼统**：日志里只有「提交回传」，看不出回传到哪个系统，也分不清首次提交与异常行「重新回传」。
3. **采集计划归位**：软件 UI（`Draftline.App`）的「采集计划」页不再需要，移到后台管理；且 UI 上显示的采集时间窗口是**旧的**，要以代码中最新的（采集服务实际在跑的）为准。

## 方案

- **只改显示层**优先（覆盖历史日志、风险最低），共享词表放 `Draftline.Core.Logging.LogText`，后台 Blazor 与操作员接口共用，避免两处翻译走偏。
- **两视图分流**：操作员软件只看**本人**的业务操作 + 登录/改密，精简友好（中文动作名 + 结果，隐藏 IP 与内部 payload）；后台管理保持**全量审计**（全人、全动作、全字段、可筛选）。
- **提交回传细化**：动作显示为「提交回传 → SRM」「重新回传 → EBS」（目标由流程推导，重传标记落 payload）。
- **采集排程单一数据源**：新增权威定义 `IngestionSchedules`，采集服务、客户端补拉、后台展示全部以此为准；修正挑图过时窗口；采集计划改到后台**只读**展示。

## 实现

### 一、日志中文化 + 视图分流（`6b22ebe`）

1. 新增 `src/Draftline.Core/Logging/LogText.cs`：动作英文→中文词表、payload 美化、操作员可见动作集合。
2. `src/Draftline.Gateway/Components/Pages/AdminLogs.razor`：动作/状态改中文、筛选改中文下拉；「影响」→「影响行数」并加悬停说明、0 显示为「—」；「详情」把历史英文 payload 翻译美化、空详情按动作合成一句可读描述。
3. `src/Draftline.Gateway/Endpoints/ApiEndpoints.cs` `GET /api/oplog/mine`：只返回本人业务操作 + 登录/改密，中文动作名 + 结果，不返回 IP 与内部 payload。
4. `src/Draftline.Core/Contracts/Http/HttpDtos.cs` `OperationLogEntry`：新增 `Result` 与 `FlowLabel`（App 免写转换器）。
5. `src/Draftline.App/Views/OperationLogView.xaml`：列改为 时间/操作/流程/表单名称/结果，去掉「操作电脑IP」列。
6. **修复丢失的 `POST /api/oplog`**：此前客户端行为上报 POST 到一个**根本不存在**的接口，被静默丢弃；现补上，工号以令牌盖章落库。

### 二、提交回传日志细化（`bb1c468`）

1. `LogText` 新增 `SubmitTarget`/`SubmitLabel`/`IsResubmit`：目标由流程推导（核价→SRM，挑图→EBS）。
2. `ApiEndpoints.cs` `/submit`：在 4 处 Submit 日志的 payload 补「重新回传」标记，使日志可区分异常行重传（`IsExceptionRetry` 此前未落库）。
3. 操作员 `/oplog/mine` 与后台 `AdminLogs` 的动作列均改用 `SubmitLabel`。正常提交 payload 不变，`audit/exists` 走结构化列、补拉判据不受影响。

### 三、采集计划移入后台 + 排程统一（`01caa9e`）

1. 新增 `src/Draftline.Core/Schemas/IngestionSchedules.cs`：采集排程**唯一权威定义**（含触发时刻）。
2. `src/Draftline.Core/Schemas/CollectionSchedules.cs`：改为从 `IngestionSchedules` 投影（客户端补拉/批次键用）。
3. `src/Draftline.Gateway/DataIngestionService.cs`：改为遍历 `IngestionSchedules.All`，窗口解算复用 `CollectionWindow.Resolve`（运行行为不变，仅去重定义）。
4. **修正挑图午间日界 15:30→14:30**：上午批 `09:31~14:30`（触发 15:00）、下午批 `14:31~18:00`（触发 18:30）。此前采集服务用 14:30、客户端补拉却用旧的 15:30，导致补拉窗口与实际采集批次对不上——现统一后一致。
5. 新增后台只读页 `src/Draftline.Gateway/Components/Pages/AdminSchedules.razor`（`/admin/schedules`）+ `AdminLayout.razor` 导航入口，展示真实触发时刻与窗口。
6. 移除 WPF 采集计划：删 `ScheduleView`/`ScheduleViewModel`，摘除 `ShellViewModel` 导航项、`INavigationService.ToSchedule`、`App.xaml` 模板。
7. `tests/Draftline.Tests/Schemas/CollectionSchedulesTests.cs`：更新至权威窗口值（该测试原本锁定的正是旧值，反证了过时问题）。
