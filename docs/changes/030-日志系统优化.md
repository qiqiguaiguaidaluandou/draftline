## 一、现状与问题

物理上只有一张 `ActivityLogs`（`src/Draftline.Gateway/Stores/ActivityLog.cs`），靠 `Action` 字符串列区分。写入点散在 6 处、约 12 段 `new ActivityLog{...}`：

| 来源 | Action |
|---|---|
| `ApiEndpoints.cs` | `Login`/`ChangePassword`/`UpdateRow`/`Suspend`/`Resolve`/`RefetchDrawing`/`Submit` |
| `PgAuditStore.cs` | `Submit`（唯一填 `WindowStart/End/AuditId` 结构化列） |
| `AdminAuthEndpoints.cs` + `AdminService.Log()` | `AdminLogin`/`Admin*` |
| `DataIngestionService.cs` | `Ingest`（`EmployeeId="SYSTEM"`） |
| `PgOperationLogStore.cs` | `Behavior`（客户端 `/oplog` 埋点） |

查看侧：操作员 `/oplog/mine`（白名单 `LogText.OperatorActions`、只看本人，**直接查 `ActivityLogs` 表**，不经 `IOperationLogStore`）；管理员 `/api/admin/logs`（全量含 IP/Payload）。技术诊断日志走独立 `ILogger`（控制台），与业务日志分离——不动。

**问题清单：**

1. **`Action` 全是散落的魔法字符串**。写入点、清理条件（`DataIngestionService` 手写 `x.Action == "Submit"`）、查询、`LogText` 各写一遍，拼错不报错，无单一源。
2. **客户端 `Behavior` 埋点与服务端动作日志高度冗余**。点"提交回传"/"重新获取图纸"客户端发一条 `Behavior`，服务端 `/submit`、`/refetch-drawing` 又各记一条权威日志——同一动作两条。且 `Behavior` **无读者依赖**：`/oplog/mine` 白名单明确排除它，仅管理员全量视图会看到，纯噪音。
3. **`Payload` 自由拼接英文 + 事后字符串替换"美化"**。写入端拼 `"Row: {x}, Reason: {y}"`、`"empId=…, admin=True"`，读取端靠 `LogText.Prettify` 连环 `.Replace("empId=","工号=")…` 反解析——脆弱（正文含 `id=`/`, ` 会被误替换），英文键→中文纯历史包袱。
4. **死代码**：`OperationLogEntity`+`operation_logs` 表+migration `AddOperationLogs`（未进 DbContext）；`FileOperationLogStore`+`OperationLogOptions`+appsettings `"OperationLog"` 节（DI 未注册）。
5. **列被借用**：`Behavior` 把页面名塞进 `BatchId` 列（`PgOperationLogStore.cs:20`），语义错位。
6. **每条日志信息不足**（详见第三节表）：`UpdateRow` 只记行键、连改了哪个字段都没有；`Suspend/Resolve/Refetch` 白丢了手上的物料名/编码；`Ingest` 无 payload。

## 三、每条日志的"补细节"清单（阶段 3.5）

原则：凡业务动作，一律带上**人可读的物料名/编码**（不只内部 `RowKey`）。

| 动作 | 现在 Payload | 补后（中文键，示例） | 数据来源 |
|---|---|---|---|
| **UpdateRow** ⚠️最弱 | `Row: {key}` | `数据行={key}；物料={名称}；修改字段=单价(10→12)、数量(5→8)` | `req.Values`（新值）+ store 回传旧值 |
| **Suspend** | `Row: {key}, Reason: {因}` | `数据行={key}；物料={code} {名称}；原因={因}` | `req.MaterialCode`/`DisplayName`（现已丢弃） |
| **Resolve** | `Row: {key}` | `数据行={key}；物料={名称}；方式=补回传成功`（区分补回传/撤销） | 端点读到的异常实体 |
| **RefetchDrawing** | `Row: {key}, Files: …` | `数据行={key}；物料={code}；新增图纸=a.pdf、b.pdf` | `req.MaterialCode` + fetched |
| **Submit** | `Target: SRM`(+失败明细) | `回传目标=SRM；成功={n}/{总}行；组={group}`（失败明细保留） | `successCount`/`registry.TotalRows` |
| **Ingest** | 无 payload | `窗口={起}~{止}；取数={n}行`（空窗记 `空窗自动完结`） | 取数窗口 + `rows.Count` |
| **Login/ChangePassword** | 成功空/失败记原因 | 失败原因保留；成功可留空（本就够） | — |
| **Admin\*** | `empId=…, admin=True` | `工号=…；管理员=是`（转中文键） | 已较全 |

读取侧顺带：操作员 `/oplog/mine` 增加"详情"列展示上述中文明细（物料、改了什么）；IP 仍只在后台全量视图。

### 异常重新回传链路：去掉重复的 UpdateRow

一次"异常处理页 → 重新回传成功"当前依次打 4 个接口、写 **4 条**日志：`/submit`(Submit) + `/batch/update-row`(UpdateRow) + `/batch/resolve-exception`(Resolve) + `/oplog`(Behavior)。

- 阶段 2 去掉 Behavior。
- **决定：此链路不再单独写 UpdateRow**——只保留 **Submit（回传审计）+ Resolve（异常闭环）**两条，行改动并入 Resolve 的 payload。
- 实现：`UpdateRowRequest` 增 `IsExceptionResolve` 标记；`/batch/update-row` 见此标记时**照常写数据、但不写 UpdateRow 日志**（改动 diff 随该次调用带回，落到随后的 Resolve 日志 payload）。**普通批次作业页的改行编辑不受影响，仍写带完整 diff 的 UpdateRow。**
- 效果：异常重新回传成功从 4 条 → **2 条**（Submit `重新回传→SRM`、Resolve `补回传成功；物料=…；修改字段=单价(10→12)`）。
