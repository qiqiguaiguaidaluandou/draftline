├─ src/TZHJ.App/Services/SessionSyncScheduler.cs  ★新增（会话内定时器，登录后起、登出停；调 BatchSyncService）
├─ src/TZHJ.Core/Contracts/IAuditGateway.cs   ★新增（第5个网关契约）
├─ tests/TZHJ.Tests              ★新增 (net8.0，跨平台，Linux 上即可跑)
│   └─ Storage/Schedules/Sync 的单元/集成测试（占测试大头）
└─ tests/TZHJ.App.Tests          ★新增 (net8.0-windows，仅 Windows 跑)

**为何拆两个测试工程**：`RowViewModel` 引用了 `System.Windows.Visibility`（操作列按钮可见性），
属 WPF，测试它的工程必须 `net8.0-windows`、只能在 Windows 跑。而 `LocalBatchStore`/`ExcelGridIO`/
`CollectionSchedules`/`BatchSyncService` 都在 Core/Infrastructure（`net8.0`、无 WPF），**现在就能在 Linux 上跑**。
故拆分：跨平台的放 `TZHJ.Tests`（占大头、CI 友好），WPF 耦合的少量放 `TZHJ.App.Tests`。

**为何把补拉编排抽到 `BatchSyncService`（Infrastructure）而非留在 ViewModel**：①手动补拉（`BatchListViewModel`）
与登录补拉（②新增）要用**同一套**"选已关闭窗 → 本地有则跳 → 审计命中则跳 → 否则取数落本地"逻辑，抽出来避免两份；
②放 Infrastructure（无 WPF）后这套核心编排**可跨平台单测**（注入假网关/假存储），是 ② 测试的重点之一。
`BatchListViewModel.ManualFetch`/`RecentWindows` 改为调用该服务（行为不变，刚修的"仅已关闭窗"语义保留）。

## 5. 边界逻辑设计

### 5.1 FileSystemWatcher 同步（文件夹即真相源）

**目标**：操作员或外部在资源管理器里增/删/改名批次目录，**列表能反映**（无需手动刷新）。

**设计**：`BatchListViewModel` 在 `LoadAsync` 后对 `LocationRootPath` 起一个 `FileSystemWatcher`
（`NotifyFilter = DirectoryName`，监 `Created`/`Deleted`/`Renamed`，不递归子文件——只关心批次目录本身的增删）。
- **去抖**：文件操作常成串到达，用一个 ~300ms 的去抖计时器合并，到点再 `await LoadAsync()`。
- **线程**：watcher 回调在后台线程，刷新要 `Dispatcher` 调度回 UI 线程。
- **生命周期**：进入页面订阅、离开/`Dispose` 时停掉（`EnableRaisingEvents=false` + Dispose），避免泄漏与重复刷新。

**测试**：watcher 涉及真实文件系统事件 + WPF Dispatcher，**不做自动化单测**（不稳定）；
以①的 curl 同级别的"手动验收"覆盖（§7）。可测的是其触发的 `LoadAsync` 本身（已在 4.1 间接覆盖）。

### 5.2 删除批次目录（甲案：如实同步、不阻止、不做持久作废标记）

- **列表层**：批次目录被删 → watcher（5.1）触发刷新 → 该批次从列表消失。不弹阻断、不留"已作废"标记。
- **作业页**：正在作业的批次目录被删 → `GetBatchAsync` 返 null → 已有逻辑提示「批次不存在或已被删除」并返回列表（保留，本期确认其在 watcher 场景下也成立）。
- 不引入任何持久化的"作废/墓碑"记录——与"文件夹即真相源"一致：没了就是没了。

### 5.3 提交闸门（已实现，本期补测 + 可选加固）

- 现状：`Batch.CanSubmit` = `Rows.Count>0 && 无 Pending 行`；`BatchWorkViewModel` 据此禁用「上传」并显示 `GateText`。
- 本期：4.1 已锁定 `Batch.CanSubmit` 判定；**可选加固**——在 `Submit()` 执行前再断言一次闸门
  （防御 UI 状态与模型短暂不一致的极端情况），不满足则提示并中止。（决策 D-可选，默认做这道二次校验。）

### 5.4 登录补拉 + 查后端审计日志（本期大头）

**要解决的核心判据**：本地某窗口没有批次时，到底"漏拉了"还是"早就处理完并清掉了"？
—— 查后端审计：**审计命中=已成功回传过 → 不补拉**（避免把已完成的窗重新拉回来甚至重传）；
**未命中 + 本地无 → 补拉**。

**新增第 5 个网关契约 `IAuditGateway`**（纯新增，不动既有四个）：
```csharp
public interface IAuditGateway {
    Task<AuditExistsResponse> ExistsAsync(FlowType flow, DateTime windowStart, DateTime windowEnd, CancellationToken ct = default);
}
```
- `MockAuditGateway`：恒返回 `exists=false`（离线模式下"本地无即补"，行为合理）。注册进 `AddTzhjMockInfrastructure`。
- `HttpAuditGateway`：`GET /api/audit/exists?flow=&windowStart=&windowEnd=`（窗口起止经 `:O` 往返，与①一致；身份以令牌为准）。注册进 `AddTzhjHttpInfrastructure`。
- `AuditExistsResponse` DTO 已在 `TZHJ.Core/Contracts/Http`（①已建），直接复用。

**`BatchSyncService.SyncAsync(flow, empId, now)`（Infrastructure，§3）逐已关闭窗**：
```
对每个已关闭窗 w：
  若 BatchExists(flow, emp, w) → 跳过（本地已有）
  否则 await audit.ExistsAsync(flow, w):
       命中 → 跳过（已处理过，不重拉）
       未命中 → FetchBatchAsync(w) 成功则 WriteFetchedBatchAsync 落本地
返回新增批次数
```

**三种触发点（都复用同一个 `SyncAsync`，故"永远不漏窗"）**：
- **手动补拉**：`BatchListViewModel.ManualFetch` 改为调 `SyncAsync(当前流程)`（行为=现在 + 审计判据）。
- **登录补拉（决策 D5）**：登录成功后，对操作员 `AllowedFlows` 的各流程各调一次 `SyncAsync`——补**离线期间**关闭的窗。
  **默认在登录完成、进入主界面后于后台带进度运行**（不卡登录窗；失败只提示不阻断，"命令型操作各自报错"——见 `NavigationService` 注释）。
- **会话内定时触发（§5.5）**：登录态下，窗一到触发时刻就自动 `SyncAsync`——补**在线期间**新关闭的窗。

**测试**：`BatchSyncService` 的全部判据走 4.1（注入假 `ILocalBatchStore`/`IDataGateway`/`IAuditGateway` + 固定时刻），
是本期最有价值的逻辑测试。Http/Mock 审计网关的线缆行为，后端侧①已 curl 验过，客户端侧随真机验收（§7）。

### 5.5 会话内定时自动取数（登录态触发）

**目标**：一直登录着、某个窗到了触发时刻（窗口关闭后≈1min），**自动取数落本地**，无需手动点更新。
这是 `CollectionWindow.TriggerTime`/`采集计划`页"登录态触发"承诺的真正落地——此前只有展示文字、没有代码兑现。

**新增 `SessionSyncScheduler`（App，单例，登录后起、登出/退出停）**：
- 内部一个 `DispatcherTimer`，**定时轮询**（决策 D10，默认 **120s** 一跳）调
  `BatchSyncService.SyncAsync` 处理操作员 `AllowedFlows` 各流程。`SyncAsync` 幂等（本地已有/审计命中即跳），
  故轮询安全：只有真到了触发时刻的新窗才会被拉。
- **为何用轮询而非"算到下一触发点单次唤醒"**：轮询对**笔记本休眠/唤醒、系统时钟改动、漏 tick、配置改时间窗**都鲁棒；
  代价仅是每 120s 一次廉价的本地 `BatchExists` 检查（绝大多数跳空）。窗关闭后最多 ~120s 内被拉到，对分钟级时间窗足够及时。
- **并发护栏**：用一个 `SemaphoreSlim(1)`/忙标志，确保定时跳与手动补拉、上一跳不重叠；忙则本跳跳过。
- **拉到新批次后（决策 D11）**：弹 **Toast** 提示（如「新批次已到：核价·上午批」）并刷新当前批次列表（若正停在该列表），
  **不打断**操作员当前作业。无新批次则静默。
- **生命周期**：登录成功后启动（与登录补拉同源，建议登录补拉跑完即转入定时轮询）；主窗关闭/登出时 `Stop`+`Dispose`。
- **承载位置**：登录后长存、跨页面存活，挂在 `ShellViewModel`（或登录成功后由 `App` 启动的单例服务）。

> 与"审计命中即跳"的小注意：某已关闭窗若本地无、审计命中（已处理过），定时轮询每跳都会再查一次该窗的
> `/audit/exists`（结果恒命中、永远跳过）。可选优化：`SyncAsync` 内对"本会话已判定跳过"的窗做记忆，免重复查审计。默认先不优化（每 120s 一次 HTTP 可接受）。

**测试**：`SessionSyncScheduler` 本身是"定时器→调服务→弹 Toast"的薄壳（含 WPF/Dispatcher），不做自动化单测；
其调用的 `SyncAsync` 判据已在 4.1 全覆盖。定时行为随真机验收（§7）。

## 7. 开发步骤（建议顺序）与验收
