## 前提认知

"没有真实接口"只挡住一件事：**后端防腐层里真正调 EBS/PLM/SRM 的那几行**。
除此之外系统的大半都能继续做——客户端、本地状态机、HTTP 链路、测试、发布配置全不依赖外部接口。
有些事现在做反而最划算（见 ①，把未来"换接口"的风险提前拆解）。

## ① 最高杠杆：提前演练"换接口"这一步（后端骨架 + 真 HTTP 链路）

**目标**：把未来"Mock → Http 大爆炸式切换"拆成已完成的小事。届时改动只剩后端防腐层内部一层，
客户端↔后端的 HTTP / 序列化 / 错误处理 / 超时届时已全部跑通验证。

- [x] 起 **ASP.NET Core 无状态后端骨架**工程（纳入 `TZHJ.sln`）—— `src/TZHJ.Gateway`
- [x] 定义四个端点的请求/响应契约，**直接复用 `TZHJ.Core` 已有形状**（`FetchRequest`/`FetchResult`/`SubmitRequest`/`SubmitResult` 等）：
  - [x] 取数端点（对应 `IDataGateway`，两阶段：`/fetch` 行+图纸元数据、`/drawings` 流式字节）
  - [x] 回传端点（对应 `ISubmitGateway`，`/submit` + 记审计）
  - [x] 认证端点（对应 `IAuthGateway`，`/auth/login` 占位发令牌）
  - [x] 配置下发端点（对应 `IConfigGateway`，`/config` 身份以令牌为准）
- [x] 把现有 **Mock 造数逻辑搬到后端**（`AntiCorruption/FakeDataSource`，确定性种子改稳定哈希，无状态可重生）
- [x] 客户端写出 `HttpDataGateway` / `HttpSubmitGateway` / `HttpAuthGateway` / `HttpConfigGateway`（`TZHJ.Infrastructure/Gateways/Http/`）
- [x] 在 `DependencyInjection.cs` 新增 `AddTzhjHttpInfrastructure`（与 `AddTzhjMockInfrastructure` 并存，`App.xaml.cs` 按 `UseMock` 切换），**UI 不动**
- [ ] 起后端 PostgreSQL：配置表 / 权限工号映射表 / **回传审计日志表**（**后置**：本期审计表**结构已在 `Stores/AuditRecord` 定下**，存储先用内存；落 PG 见开发文档①§8）

> **配置下发端点 / 配置表的作用**：登录时客户端调 `IConfigGateway.GetConfigAsync(工号)` 拿回一组
> `ClientConfig` 开关——本地根目录、后端网关地址、**时间窗规则**（核价2窗/挑图3窗）、**表单字段集**
> （作业网格显示哪些列、哪列待填）、已处理批次保留天数。配置表把这些值存在后端，**管理员改一处、
> 所有客户端下次登录生效，无需重打包 ClickOnce 逐台更新**。这正是方案"时间窗可配置、字段配置化、
> 加字段不改代码"的落地处（作业网格的列由字段 schema 动态生成）。
> 边界：配置只负责"显示这列"，若该列要显示新的 EBS/PLM 数据，仍需防腐层能取到值；纯手填的待填列则靠配置即可加。
> 现在客户端**已在消费 `ClientConfig`**（Mock 返回默认值），把它改走真 HTTP + 配置表，等于提前验证
> "改配置即生效、不重发"这一承诺——可先用简单表或 JSON 起步，后续再正规化。

## 建议开发顺序

1. **从 ① 起步**：杠杆最高，把最大的未来风险（接口切换）现在就拆掉。
2. **② 的测试穿插补**：边做边给核心逻辑兜底，后续重构/换接口才有信心。
3. **③ 随时插入**：独立小项，不挡主线。
4. **④ 尽早在 Windows 上验收**：并行进行，越早越好。
