# 改代码后重新部署速查指南

> 面向「改完代码，怎么重新发版」的日常场景。真实路径/命令与完整背景见
> `部署与发布实操手册(runbook).md`；本文是其中「日常发版」部分的精简速查 + 关键概念解释。

---

## 0. 大前提：前后端是两条独立发布线

| 你改了 | 操作 | 影响面 |
|---|---|---|
| **后端**（`Draftline.Gateway`） | Linux 服务器上重发 + 重启 | **客户端完全不用动**，全员即时生效 |
| **客户端**（`Draftline.App`） | Windows 上出 ClickOnce 包 + 上传 | **后端不用动**，用户下次启动自动更新 |
| **共享库**（`Draftline.Core` / `Draftline.Infrastructure`） | 看哪边用到就做哪边 | 两边都用到 → **两边各做一次** |

> ⚠️ 改后端只重部署服务器、客户端无感；改客户端只重发 ClickOnce、后端不动。两条线各走各的。

关键路径对照（运行身份 `jptadmin`）：

| 变量 | 实际值 |
|---|---|
| 后端部署目录 | `/home/jptadmin/kqspace/draftline-all/draftline-gateway` |
| ClickOnce 分发目录（`DistPath`） | `/home/jptadmin/kqspace/draftline-all/clickonce`（与后端目录**平级、解耦**） |
| 访问前缀（`RequestPath`） | `/draftline` |
| 后端监听 | `http://0.0.0.0:8080` |

## 1. 后端改动后重新部署（在 Linux 服务器上）

```bash
# 1) 重新发布（覆盖式，绝不要先 rm -rf 整个目录）
dotnet publish src/Draftline.Gateway/Draftline.Gateway.csproj -c Release \
  -o /home/jptadmin/kqspace/draftline-all/draftline-gateway

# 2) 有新 EF 迁移才跑（没有就跳过）
dotnet ef database update --project src/Draftline.Gateway/Draftline.Gateway.csproj

# 3) 重启服务
sudo systemctl restart draftline-gateway
sudo systemctl status draftline-gateway          # active (running) 即成功

# 4) 验证
curl -i http://localhost:8080/                   # 本机自测
sudo journalctl -u draftline-gateway -f          # 应有 "Now listening on: http://0.0.0.0:8080"
```

**注意坑：**

- 重发后确认 `appsettings.local.json` 仍在部署目录（真实配置/密码/连接串都在这个 gitignore 的文件里，不进仓库）：
  ```bash
  ls -l /home/jptadmin/kqspace/draftline-all/draftline-gateway/appsettings.local.json
  ```
- **绝不要** `rm -rf <部署目录>/*` 或 `rsync --delete`——会误删 `App_Data`、`clickonce`、`appsettings.local.json`。用普通覆盖式 `publish -o` 就安全。
- 若该机 `dotnet` 不在 PATH（装在 `~/.dotnet`），用绝对路径 `~/.dotnet/dotnet publish ...`。

## 2. 客户端改动后重新发布（必须在 Windows 上）

> ⚠️ ClickOnce 包**只能在 Windows 出**，Linux 会报 `MSB3096 GenerateLauncher only supported on Windows`。

**① 版本号 +1（最关键，不加则永远不更新）**

编辑 `src\Draftline.App\Properties\PublishProfiles\ClickOnceProfile.pubxml`：
```xml
<ApplicationVersion>1.0.0.2</ApplicationVersion>   <!-- 上次 1.0.0.1 → 本次末位 +1 -->
```
> 这是客户端识别「有新版」的**唯一依据**。

**② 清理旧产物（切过配置/加过 RID 后必做一次，详见 §4）**
```powershell
Remove-Item -Recurse -Force src\Draftline.App\bin, src\Draftline.App\obj -ErrorAction SilentlyContinue
Remove-Item -Recurse -Force src\Draftline.Core\bin, src\Draftline.Core\obj -ErrorAction SilentlyContinue
Remove-Item -Recurse -Force src\Draftline.Infrastructure\bin, src\Draftline.Infrastructure\obj -ErrorAction SilentlyContinue
```

**③ 发布（用命令行 msbuild，别用 VS 图形界面——它会吞掉错误）**

开 **「Developer PowerShell for VS 2022」**，仓库根目录跑：
```powershell
msbuild src\Draftline.App\Draftline.App.csproj "/t:Restore;Publish" /p:PublishProfile=ClickOnceProfile /p:Configuration=Release /v:m
```
> 含分号的参数**必须带双引号** `"/t:Restore;Publish"`，且 `Restore` 与 `Publish` 必须同一次调用（否则还原不带 `win-x64` → NETSDK1047）。

产物在 `src\Draftline.App\bin\publish\`（含 `setup.exe`、`Draftline.App.application`、`Application Files\`）。

**④ 覆盖上传到分发点**（= `DistPath`，是后端目录**外面平级**的 `clickonce`，**不是**后端目录下）
```powershell
cd C:\...\src\Draftline.App\bin\publish
scp -r * jptadmin@<后端IP>:/home/jptadmin/kqspace/draftline-all/clickonce/
```
> 先 `cd` 进 publish 再 `scp -r *`，正确带过含空格的 `Application Files`。**保留旧版本文件夹别删**——只新增新版本目录 + 覆盖顶层 `.application`。

**⑤ 验证**（任意能连后端的机器）：
```
http://<后端IP>:8080/draftline/Draftline.App.application
```
能下到 XML 即通 ✅（404 多半是改过 `DistPath` 后后端没重启，`sudo systemctl restart draftline-gateway`）。

用户侧无需操作——下次从快捷方式启动即自动检测新版、先下载安装再进入。

## 3. 换服务器地址 / 上 HTTPS 时

以下地址会**烤进 ClickOnce 部署清单**，改了必须重发客户端包：

- `ClickOnceProfile.pubxml` 的 `<PublishUrl>` / `<InstallUrl>`
- `src\Draftline.App\appsettings.json` 的 `Http:BaseUrl`（客户端连后端的基址）

## 4. 附：什么是「旧产物」，为什么要清

「旧产物」指的是 **`bin/` 和 `obj/` 这两个编译目录里，上一次构建残留下来的文件**——都是从源码自动生成、可随时重建的东西，不是手写代码。

| 目录 | 内容 |
|---|---|
| **`obj/`** | 编译**中间产物**：NuGet 还原记录（`project.assets.json`，记录本次还原的目标框架/RID）、编译临时文件；**WPF 特有**：XAML 编译成的 `.baml`、`.g.cs` 生成代码 |
| **`bin/`** | 编译**最终输出**：生成的 `.dll` / `.exe`、依赖库、`appsettings.json` 副本等 |

**为什么客户端发布前要清（后端不用）：**

`obj/project.assets.json` 会**缓存上一次还原的配置**。本项目客户端踩过一个实测坑：

1. 项目后来加了 `<RuntimeIdentifiers>win-x64</RuntimeIdentifiers>`（为修 ClickOnce 发布的 NETSDK1047）。
2. 加 RID 前后，WPF 输出目录结构变了——带 RID 时路径多一层 `win-x64`。
3. 若 `obj/` 还留着**加 RID 之前**的缓存，编译器按旧路径找 `FluentTheme.baml`，报：
   ```
   找不到 ...\Styles\FluentTheme.baml   （路径缺 win-x64 层）
   ```

删掉 `bin/obj` 强制**从零重新还原 + 编译**，绕开新旧混淆。

**为什么要连 Core/Infrastructure 一起清：** 因为 `Draftline.App` 依赖这两个库，只清 App、依赖库 `obj` 还是旧缓存，一样会带进错误的目标框架/RID 信息，所以三个工程一起清。

> 一句话：旧产物 = 上次编译留下的、可自动重建的 `bin`/`obj` 缓存；清它是为避免「加了 win-x64 后旧缓存指错路径」这个 WPF ClickOnce 专属坑。此步只在**切换过配置**（加 RID、切 Debug/Release）后需做一次，稳定后日常发版可省，为保险仍列进标准流程。

## 5. 发版检查清单（Cheat Sheet）

**后端发版：**
- [ ] `dotnet publish -o 部署目录`
- [ ] `appsettings.local.json` 在位、`Urls`/连接串正确
- [ ] 有迁移则 `ef database update`
- [ ] `systemctl restart draftline-gateway` → `status` 为 running
- [ ] `curl http://<后端IP>:8080/` 通

**客户端发版：**
- [ ] `ApplicationVersion` 末位 +1
- [ ] pubxml `InstallUrl` / appsettings.json `BaseUrl` 指向对的后端地址
- [ ] 清 obj/bin
- [ ] `msbuild "/t:Restore;Publish" ...` 成功
- [ ] `publish\` 内容上传到 `DistPath` 目录（不是后端目录下）
- [ ] `http://<后端IP>:8080/draftline/Draftline.App.application` 能下到 XML
- [ ] 客户端 `setup.exe` 安装、登录、取数验证
