# 图纸核价 / 挑图纸系统 — 方案设计

> 本文档是需求与实现方案的基线，后续讨论在此基础上修改。
> 最后更新：2026-05-22

---

## 1. 业务背景

系统承载**两个相互独立的业务流程**：图纸核价、挑图纸。二者从 EBS 抓取的**源数据本就不同、必须严格隔离、绝不混为一谈**。

采用「**同一平台、两条独立数据线**」的口径（方案 A）：

- **独立的部分（按流程严格隔离，数据互不可见）：** 取数（各自的 EBS 取数接口/参数/数据集）、批次、表单字段、采集时间窗、回传目标；
- **共用的部分（只做一套，省搭建与运维）：** 后端网关、桌面客户端（内分两个独立功能区）、登录认证、权限框架、部署与自动更新。

| | 流程① 图纸核价 | 流程② 挑图纸 |
|---|---|---|
| 源数据 | 核价专属的 EBS 需求数据 | 挑图专属的 EBS 需求数据（与核价不同） |
| 操作员做什么 | 对物料核价（手填目标价，必填） | 填"是否机加中心可以做"（是/否，必填） |
| 回传到 | SRM | EBS |
| 采集时间窗 | 昨天15:31~今天09:30、09:31~15:30（2 窗） | 昨天18:01~今天09:30、09:31~15:30、15:31~18:00（3 窗） |
| 批次 | 只装核价源数据 | 只装挑图源数据（与核价批次永不混） |

**每个流程各自的主干（流程内一致）：**
1. 按本流程某时间窗，向 **EBS** 取**本操作员（按工号等标识限定）**的需求数据（物料清单）；
2. 按物料编码到 **PLM** 拉取每个物料对应的**图纸附件 + PLM 侧数据（如"是否存在变更"状态）**；
3. 把「需求数据 + 这批所有图纸」组织成一个**本流程的批次**，下发并**落到操作员本地**，置为"待处理"；
4. 操作员在软件里打开本地表单作业；**图纸不在软件内查看**，操作员自行到本地批次文件夹用本机软件（PDF 阅读器 / CAD）打开；
5. 整批处理完成后提交，经后端回传到本流程的目标系统，回传成功后批次转"已处理"。

**涉及的外部系统：** EBS（两流程各自的需求来源 / 挑图回传）、PLM（按物料编码取图纸附件 + "是否存在变更"等物料数据）、SRM（核价回传）、DHR（后续接入，做登录认证 + 提供组织/岗位驱动权限）。所有取数/回传均经**后端网关**调用，接口由各系统方提供，本系统作为调用方。

## 2. 整体架构

**形态：后端无状态「取数/回传网关」 + Windows 桌面客户端（数据与状态都在本地） + 后端仅保留少量配置/权限/审计。** 不做纯网页、也不是把数据集中存在服务器。一个平台内承载两条独立数据线。

推导依据：
- **图纸与数据要落到操作员本地、文件夹即工作现场、图纸由操作员自行在本地查看** → 网页应用没法自由往本地建这套目录结构供操作员直接访问 → 操作端用**桌面客户端**（本地即状态的载体）；
- **凭证安全 + 接口防腐 + 统一对接外部** → 取数/回传仍**走后端**（后端持外部系统凭证、包防腐层，凭证不下发到各 PC）；
- **状态用本地文件夹位置（待处理 / 已处理）表示** → 后端**不持权威业务状态**，本地即状态。

```
        ┌──────── 后端：无状态「取数/回传网关」 ────────┐
[EBS]─核价需求(带工号)─→│                                  │
[EBS]─挑图需求(带工号)─→│  防腐层 / 认证(DHR) / 配置下发   │─回传─→[SRM] (核价)
[PLM]─图纸+变更状态────→│  回传审计日志                    │─回传─→[EBS] (挑图)
                       │  （不存图纸、不持业务状态）       │
                       └─────────────────┬────────────────┘
                              API（取数下发 / 回传）
                                          │
                                          ▼
              Windows 桌面客户端 (~10 台，每人固定一台)
  登录(DHR) →【核价功能区】/【挑图功能区】(数据互不可见)
  → 按定时规则(登录态触发 + 登录补拉)向后端取本人数据
  → 落本地：核价|挑图/<工号>/{待处理|已处理}/<取数时间范围>/...
  → 软件渲染待填表格作业（核价填目标价 / 挑图填是否机加中心可以做；软件不看图，图纸由操作员自行到本地文件夹查看）
  → 整批经后端回传 → 回传成功后批次目录 待处理 → 已处理
  状态机 + 异常待跟进池 全部在本地（文件夹位置 = 批次状态）
```

**职责划分：**
- **后端（无状态网关）**：带工号向 EBS/PLM 取数、向 SRM/EBS 回传；外部接口防腐层；认证（DHR 对接点）；配置下发（时间窗规则、接口地址、凭证）；**回传审计日志**。**不存图纸文件本体、不持业务状态。** 凭证只在后端，不下发到客户端。
- **桌面客户端（厚客户端）**：登录后按定时规则（登录态触发 + 登录补拉）向后端要**本人**数据 → 按命名/存放规则落本地 → 渲染待填表格作业（核价填目标价 / 挑图填是否机加中心可以做；**软件不内嵌看图**，图纸由操作员自行到本地批次文件夹用本机软件查看）→ 整批经后端回传 → 回传成功后把批次目录从 `待处理` 移到 `已处理`。**状态机与异常待跟进池都在本地。**
  - **软件视图 = 本地文件夹的视图**：批次列表/状态直接**映射本地 `待处理/已处理/批次` 文件夹**（文件夹即真相源，软件读取并展示，不在软件内另存一份清单）；并可**一键在资源管理器中打开**对应批次目录，直接访问原始图纸文件。
  - **同步与防御**：用 `FileSystemWatcher` 监听文件夹变化、外部改动即重扫刷新；校验批次目录完整性（表格在否、图纸缺否）并提示；**`待处理→已处理` 仅由"回传成功"驱动**，手动拖动不作数，检测到不一致要纠正/告警（因后端不存文件，本地误删的图只能重拉 PLM 且可能版本漂移——见已接受的取舍）。
  - **本地删除批次目录如何处理（甲案：如实同步 + 警告，不做持久"作废"标记）**：批次目录被外部删除时，软件重扫后该批次即从列表消失，**不假装它还在**（文件夹即真相源）。后果分两种：
    - 删**「待处理」**批次 = 未提交的暂存进度 + 图纸**全部丢失、无处恢复**（后端没存、还没回传）；且**只要采集窗口仍在补拉规则覆盖范围内，下次登录/取数会把它当"漏数"重新拉成一个空白批次**（已填值不会恢复），等窗口过期出了补拉范围才真正消失。故删除/检测到时**弹警告告知后果**，但不阻止、也不持久记住"别再拉"（如需"删了不再回来"须给后端加作废标记，本期不做）。
    - 删**「已处理」**批次 = 只丢本地副本，结果已在 SRM/EBS、后端有回传日志，不影响对账；**且不应触发补拉**（靠"补拉前查后端回传日志"保证，见 §6）。

### 数据流 / 客户端取数模型

**核心：客户端为厚客户端，数据与状态都在本地；后端只是带凭证的取数/回传通道，本身不留数据、不持状态。**

登录后一次典型作业的链路：

1. **登录** → 经后端做认证（DHR/占位），拿到身份与可进入的功能区（流程权限）。
2. **取数（定时规则 + 登录态触发 + 登录补拉）** → 客户端在登录状态下，按配置的时间窗规则触发取数；**登录时先补拉**按规则现在本该有、但本地还没拉到的窗口，避免漏数。取数请求经后端，后端**带工号**调 EBS（取本人需求）+ PLM（按物料编码取图纸 + "是否存在变更"等 PLM 侧数据），把「需求 + 图纸 + PLM 数据」组织成批次**下发到本地**。
3. **落本地** → 按命名/存放规则写入本地（见 §4），批次进 `待处理` 文件夹。
4. **作业** → 客户端渲染待填表格供填写（核价填目标价、挑图填是否机加中心可以做），逐条暂存先写本地；**图纸不在软件内看**，操作员需要时自行到本地批次文件夹用本机软件打开（软件提供"在资源管理器中打开本批次目录"入口便于定位）。
5. **整批提交** → 客户端调后端回传 API，**回传由后端执行**（核价→SRM、挑图→EBS），客户端不直连外部系统；后端记一条回传审计日志。
6. **状态流转** → 后端确认回传成功后，客户端把该批次目录从 `待处理` 移到 `已处理`（即"状态变化 = 文件夹变化"）。

**已接受的取舍（选"本地即状态"换来的简单，代价如下，均为有意识选择）：**
- **不跨机器续作**：数据在哪台机器就只能在哪台干 → 前提是**每个操作员固定用同一台机器**；
- **异常池每台机器各管各的**，不跨机器汇总；
- **可追溯只靠后端回传审计日志**（本地文件被删即无痕）→ 因此后端**必须**为每次回传留日志，这是唯一的审计抓手（见 §4、§8）。

### 表单作业与上传（操作员核心交互）

核价/挑图的核心动作就是**直接在表格上填**，全部在本系统内完成（不在 Excel 里改）。

**表格从哪来：** 取数分两步——后端先调 **EBS 取带料号的需求数据**（= 表格的行 + EBS 侧只读字段），再**按料号调 PLM 取图纸 + PLM 侧数据**（含"是否存在变更"，关联到对应行）。客户端加**本流程的待填列**（核价=目标价列、挑图=是否机加中心可以做列），待填列按流程配置、加字段不改代码。

**核价表单字段（当前已知，后续可增；字段配置化、加字段不改代码）：**

| 字段 | 来源 | 可编辑 | 回传 SRM 校验 |
|---|---|---|---|
| 物料编码 | EBS | 只读 | — |
| 型号 | EBS | 只读 | — |
| 名称 | EBS | 只读 | — |
| 需求数量 | EBS | 只读 | — |
| 当前是否存在变更 | PLM（按物料编码取） | 只读 | — |
| **目标价** | 操作员手填 | **可编辑** | **必填：回传 SRM 前校验，未填不得提交该行** |

> 目标价是核价流程的待填列、**必填项**，**完全由操作员手填、不从 EBS 取**（取数时该列为空，等操作员填）：整批提交时若有正常行未填目标价，提交闸门不通过（该行须先填目标价或挂起异常）。字段后续会增加，以上为首批已知字段。

**挑图表单字段（当前已知，后续可增；字段配置化、加字段不改代码）：**

| 字段 | 来源 | 可编辑 | 回传 EBS 校验 |
|---|---|---|---|
| EBS-ID | EBS | 只读 | —（行标识 / 回传关联键，见 §9） |
| 库存组织 | EBS | 只读 | — |
| 来源单号 | EBS | 只读 | — |
| 项目 | EBS | 只读 | — |
| 产品线 | EBS | 只读 | — |
| 方案编号 | EBS | 只读 | — |
| 部门描述 | EBS | 只读 | — |
| 物料编码 | EBS | 只读 | — |
| 物料描述 | EBS | 只读 | — |
| 是否存在变更 | PLM（按物料编码取） | 只读 | — |
| 当前数量 | EBS | 只读 | — |
| 创建日期 | EBS | 只读 | — |
| 需求日期 | EBS | 只读 | — |
| 申请人名称 | EBS | 只读 | — |
| 备注 | EBS | 只读 | — |
| **是否机加中心可以做** | 操作员手填 | **可编辑（是/否）** | **必填：回传 EBS 前校验，未填不得提交该行** |

> 是否机加中心可以做是挑图流程的待填列、**必填项**，取值 **是 / 否**（下拉），**完全由操作员手填、不从 EBS 取**：整批提交时若有正常行未填，提交闸门不通过（该行须先填或挂起异常）。**EBS-ID** 作为行标识与回传 EBS 的关联键（须在 §9 与 EBS 确认其回传用途）。字段后续会增加，以上为首批已知字段。

**表格本体 = 本地一个 Excel（.xlsx）：**
- 每个批次目录里存**一个 `.xlsx`**：料号 + 需求字段（只读）+ 待填列；行状态/关联图纸用额外列或隐藏 sheet 记录（或另置 sidecar，详见 §4）；
- 客户端用 .NET Excel 库（**ClosedXML / EPPlus / NPOI**）读 xlsx 渲染**可编辑网格**，编辑/暂存**写回同一个 xlsx**；
- 操作员**在软件网格里填、一般不在 Excel 里改**（文件就在批次目录、随时可打开查看；被外部改动则由 FileSystemWatcher 同步）。

**软件内操作体验：**
- 每行一个料号，需求字段只读、待填列可编辑（核价目标价=数值、挑图是否机加中心可以做=下拉「是/否」）；
- **图纸不在软件内看**：操作员需要看图时，自行打开本地批次文件夹、用本机软件（PDF 阅读器 / CAD）查看；软件只提供"在资源管理器中打开本批次目录"入口便于定位（图纸文件名带物料编码前缀，便于按料号找图）；
- 待填列做**校验**（必填、数值格式/范围、下拉枚举），不合格给提示。

**上传按钮（= 整批提交回传）：**
1. **提交闸门**：批次内每行要么已填、要么挂异常，无"待处理"行时"上传"才可点（否则置灰 + 提示还差哪些行）；
2. 点击 → 客户端把**正常行**打包 → 调**后端回传 API** → 后端调对应外部系统（**核价→SRM、挑图→EBS**），客户端不直连外部；
3. 回执：成功行 → "已上传"；**挂起异常行不回传**；后端记一条回传审计日志；
4. **部分失败**：依赖回传幂等（§9 待确认），后端记录哪些行已成功、失败的可重试且不重复写；
5. 整批回传成功 → 批次目录 `待处理` → `已处理`（状态流转见 §7）。

> 回传字段按 §9 与 SRM/EBS 逐条确认（核价：物料编码+目标价（必填）+…；挑图：EBS-ID+物料编码+是否机加中心可以做（必填）+…）。

## 3. 技术选型

| 模块 | 选型 | 说明 |
|---|---|---|
| 桌面客户端 | **.NET + WPF（C#）** | Windows 原生、轻量；直接读写本地文件系统承载"本地即状态"；表格密集型表单适配好；ClickOnce 自动更新，十台机器无压力；内分两功能区；承载本地存储与状态 |
| 后端网关 | **ASP.NET Core（C#）** | 与桌面端同语言，小团队一套技能；无状态取数/回传网关：调外部接口、防腐层、认证、配置下发、回传审计日志 |
| 数据库 | **PostgreSQL**（或更轻方案） | **角色已大幅缩小**：只存配置（时间窗规则/接口地址）、权限与工号映射（DHR 前）、**回传审计日志**。不存业务数据、不存图纸 |
| 本地数据存储 | **客户端本地文件系统** | 数据 + 图纸都落本地；`待处理 / 已处理` 文件夹 + `manifest` 表示批次与状态（见 §4） |
| 定时调度 | **客户端内定时器**（取数触发）+ 后端 **Hangfire**（可选，回传失败重试/审计） | 取数定时挂在客户端登录态上（按配置规则 + 登录补拉）；回传重试与日志归档可放后端 Hangfire |
| 认证/权限 | 先自建用户/角色/工号映射，DHR 后续接入 | 见 §5 |

> **后端语言已定：C#（ASP.NET Core）**，即全 C# 方案——桌面端与后端同一门语言，小团队一套技能栈、一套构建/调试链路，最省人。（曾考虑 Java/Spring Boot 岔路口，已排除。桌面端无论如何固定 WPF。）
>
> **已移除：MinIO / 集中对象存储。** 因为后端不再存图纸文件本体（本地即数据的家），对象存储无用武之地。

## 4. 数据存储设计

**数据与图纸都存在操作员本地；后端只留极少量元/配置信息。**

### 本地存储（命名 / 存放规则）

按"流程 → 工号 → 状态(文件夹) → 批次(取数时间范围) → 表格+图纸"分层，天然隔离、好定位、好清理：

```
<本地根目录>/
  核价/  |  挑图/                          ← 流程（顶层，两流程严格隔离）
    <工号>/                                ← 谁的（本机固定一个操作员）
      待处理/                              ← 刚取到、未做完的批次都在这
        <取数时间范围>/                    ← 一个批次 = 一个文件夹，命名为取数时间范围
              清单表格.xlsx                ← 一个 Excel：料号 + 需求字段(只读) + 待填列(核价目标价/挑图是否机加中心可以做) + 行状态
              <物料编码>__<原文件名>.pdf    ← 这批所有图纸，平铺在批次目录下
              <物料编码>__<原文件名>.step      （文件名加物料编码前缀防重名）
              ...
      已处理/                              ← 回传成功后，整个批次目录从"待处理"移入
        <取数时间范围>/  ...
      异常待跟进/                          ← (建议) 跨批次的异常行汇总，记来源批次
```

- **批次文件夹名 = 取数时间范围**（流程已是上层目录，名字里不再带流程）；命名需含日期以跨天唯一，如 `20260521_0931-1530`、跨夜窗 `20260520_1531-20260521_0930`；
- **每个批次文件夹里 = 一个表格 + 这批所有图纸（平铺，不再按物料分子目录）**；
- **批次状态 = 所在文件夹**：`待处理`（含"处理中"——动过但未提交）/ `已处理`（已整批回传成功）；
- 图纸文件名加**物料编码前缀**，避免不同物料同名图纸互相覆盖；
- **"一个表格" = 本地一个 `.xlsx`**：存料号 + 需求字段（只读）+ 待填列（核价=目标价 / 挑图=是否机加中心可以做）；核价只读字段首批含 物料编码、型号、名称、需求数量（EBS）+ 当前是否存在变更（PLM 按物料编码取）；挑图只读字段首批含 EBS-ID、库存组织、来源单号、项目、产品线、方案编号、部门描述、物料编码、物料描述、当前数量、创建日期、需求日期、申请人名称、备注（EBS）+ 是否存在变更（PLM 按物料编码取）；均只读、后续可增；行状态/关联图纸/来源批次用额外列或隐藏 sheet 记录（或另置 sidecar，二选一）。客户端用 Excel 库（ClosedXML/EPPlus/NPOI）读它渲染网格、编辑写回同一 xlsx；操作员在软件里填、一般不在 Excel 里改（见 §2 表单作业与上传）；
- `异常待跟进/` 为跨批次的异常行准备一个稳定位置（本机/本流程内）；
- **保留 / 清理策略**做成可配置（如"批次进已处理 N 天后清"），避免本地无限堆积；
- **软件直接以这套文件夹为模型**：批次列表即扫描这些目录得来，并提供"在资源管理器中打开"入口；文件夹对操作员可见可达，故软件需 `FileSystemWatcher` 同步 + 完整性校验 + 防误操作（见 §2 客户端职责）。

### 后端存什么

| 存什么 | 说明 |
|---|---|
| 配置 | 各流程时间窗规则、外部接口地址、调用凭证 |
| 权限 / 工号映射 | 工号 → 可进入的流程 / 角色（DHR 前自建；DHR 后改为映射适配） |
| **回传审计日志** | 每次回传记一条：谁、哪流程、哪批、哪些行、关键值、时间、成功/失败。**两个用途**：① 本地即状态模式下唯一的可追溯抓手；② **补拉时据此判断窗口是否已提交过**，避免删了「已处理」批次被误判成漏数重拉（见 §6） |

> 后端**不存图纸、不持业务状态**。文件访问全在本地（操作员本机直接读）；取数时由后端鉴权后下发。

## 5. 权限与 DHR

**认证与授权分两层设计**，以便后续 DHR 平滑接入：

- **认证（你是谁）= 身份来源**：后续由 DHR/SSO 提供（组织、岗位、工号）。现阶段先用系统自有用户表占位，**接口预留**，将来把身份来源换成 DHR 即可。
  - DHR 支持 OIDC/LDAP/CAS 则走标准协议；否则调 DHR 接口同步组织/人员；中间留映射层。
- **授权（你能看/能做什么）**：
  - **流程权限（首层隔离）**：能否进核价、能否进挑图——两流程数据互不可见，可只授权其一；
  - **功能权限（RBAC）**：流程内能否核价/填写、能否提交回传——按角色控制；
  - **数据范围**：实现"每个人获取的数据可能不同"。**本方案主要由取数接口携带工号在源头限定**——后端带该操作员工号调 EBS，源系统只返回本人对应数据。即数据边界主要由"工号取数"决定，后端只需确认该工号的流程权限。

> DHR 接入时只做"DHR 的部门/岗位/工号 → 本系统流程权限/角色"的映射适配。

## 6. 批次与采集时间窗

**批次 = 某流程一个采集时间窗、为某操作员拉到的全部数据**（落地为本地一个批次文件夹）。状态挂在批次级别（=文件夹位置），整批一起流转。**批次严格按流程隔离，核价与挑图的批次永不混。**

**核价流程时间窗：**

| 批次 | 采集时间窗 | 触发拉取时刻 |
|---|---|---|
| 上午批 | 昨天 15:31 ~ 今天 09:30 | 约今天 09:30 之后（且软件处于登录态时） |
| 下午批 | 今天 09:31 ~ 今天 15:30 | 约今天 15:30 之后（且软件处于登录态时） |

> 两窗 `15:31→09:30` 与 `09:31→15:30` 首尾相接、不重不漏铺满 24 小时（日界在 15:30/15:31）。

**挑图流程时间窗（与核价不同，三窗）：**

| 批次 | 采集时间窗 | 触发拉取时刻 |
|---|---|---|
| 夜间批 | 昨天 18:01 ~ 今天 09:30 | 约今天 09:31 之后（且软件处于登录态时） |
| 上午批 | 今天 09:31 ~ 15:30 | 约今天 15:31 之后（且软件处于登录态时） |
| 下午批 | 今天 15:31 ~ 18:00 | 约今天 18:01 之后（且软件处于登录态时） |

> 三窗 `18:01→09:30`、`09:31→15:30`、`15:31→18:00` 首尾相接、不重不漏铺满 24 小时。
> 与核价的差异：挑图为**三窗**（多一个傍晚 15:31~18:00 窗），且**日界在 18:00/18:01**（核价日界在 15:30/15:31）；两流程时间窗各自独立配置、互不影响。

**设计要点：**
- 每个流程**各自的取数接口/参数**、**各自的时间窗**，均可配置，不写死在代码里；
- 批次唯一键 = (**流程类型** + 窗口起止)，既防止重复拉取/补拉产生重复批次目录，也保证两流程隔离；
- **取数挂在客户端登录态上**：软件登录后按规则触发；**登录补拉**——一登录先把按规则现在本该有、但本地还缺的窗口补齐，再进入正常定时节奏（不再是"无人值守"，靠补拉兜底防漏）。
- **补拉前必须查后端回传审计日志**：判断某窗口是否"漏数"时，要同时看**本地有无** + **后端有无该窗口的回传记录**——只有"本地没有 **且** 后端查不到回传记录"才补拉。否则删掉一个**「已处理」**批次会被误判成漏数、重新拉成空白待处理批次，造成重复劳动甚至重复回传。（删**「待处理」**批次因从未回传、后端本就无记录，仍会被补拉回来——这是预期行为，见 §2「本地删除批次目录如何处理」。）

## 7. 状态机

两个流程各自独立运行同一套状态机逻辑，**状态全部在本地维护**（行级在 manifest，批次级用文件夹位置表示）。

**物料明细行（每条物料）：**
```
待处理 ──核价/填写──→ 已核价/已填(暂存,本地) ──整批提交,回传成功──→ 已上传
   │
   └──标记异常──────→ 挂起异常 ──(整批提交,不回传)──→ 进【本地异常待跟进池】
```

**批次（=文件夹位置）：**
```
待处理 ──(开始作业、有暂存)──→ 待处理(内含"处理中") ──(整批提交,回传成功)──→ 已处理
                                                          (目录从 待处理 移入 已处理)
```

**提交规则：**
- **提交闸门**：批次内没有"待处理"行了（每条要么已核价/已填、要么挂起异常），"整批提交"才可点；
- 支持**逐条暂存（写本地）、跨班接着干**（同一台机器），但提交是**整批一次性**；
- 点提交时：正常行 → 经后端回传本流程目标系统（核价→SRM，挑图→EBS） → 变"已上传"；挂起异常行 → **不回传**；后端确认成功后整个批次目录 → 移入 `已处理`。

## 8. 异常处理流程

挂起异常的行走一条**跨批次（但仍在本流程内、本机内）的小流程**：

```
挂起异常 ──(整批提交时不回传)──→ 进【本地异常待跟进池】(本机/本流程内、跨批次)
                                    │  记录来源批次 + 异常原因
                                    │  异常解决后补核价/补填写 + 经后端补回传
                                    ▼
                                 已上传(销账)
```

**要点：**
1. 异常行**不阻塞批次**：整批提交时正常行回传、异常行不传，批次照常移入 `已处理`；
2. 异常行**完全不回传**给目标系统（SRM/EBS 只收正常行）；
3. 异常行进**本地的"异常待跟进"池**（跨批次但不跨流程、不跨机器），仍记着来源批次（可追溯）。因此会出现"批次已移入已处理、名下仍有 N 条挂在异常池"——允许，二者独立；
4. 后续异常解决（图纸补齐/价格能定等），在异常池中补处理、单条或多条勾选经后端补回传，行变"已上传"并销账；
5. 核价、挑图各有自己的异常池，互不可见；**异常池随机器**（本地即状态），不跨机器汇总；
6. **建议**：标记挂起时记录异常原因（图纸缺失/价格待定/其他），便于分类、统计与后续跟进。（待确认是否加）

## 9. 外部接口对接

接口由各系统方提供，本系统为调用方。**"能直接用" ≠ "刚好满足需求"**，需尽早拿到接口文档逐条核对。后端对每个外部接口包一层**适配层（防腐层）**：业务逻辑只跟内部模型打交道，接口的格式怪癖挡在适配层；对每次调用记日志、失败可重试（回传步骤尤其重要）。

> 注意：核价与挑图各有自己的 EBS 取数接口/参数，对接时要分别确认。

### 对接前必须确认的清单（⭐ = 成败点）

**EBS — 取需求数据（核价、挑图各一套，分别确认）**
- ⭐ 取数携带的**标识是什么**（工号？还是别的），源系统据此**只返回本人对应数据**——这是"数据范围"的实现，须确认标识来源与传递方式；
- ⭐ 能否**按时间段查询**？（批次依赖时间窗）若仅能拉全量/按单号，则需拉全量自过滤或请其加时间参数；
- ⭐ 两个流程的源数据**用什么区分**（不同接口？同接口不同类型参数？），确保取数互不串；
- 返回字段是否覆盖物料编码 + 表单所需全部字段；**核价 EBS 取数首批需含**：物料编码、型号、名称、需求数量（**"目标价"由操作员手填、"当前是否存在变更"由 PLM 按物料编码返回**，两者均不在 EBS 取数返回字段之列）；**挑图 EBS 取数首批需含**：EBS-ID、库存组织、来源单号、项目、产品线、方案编号、部门描述、物料编码、物料描述、当前数量、创建日期、需求日期、申请人名称、备注（**"是否机加中心可以做"由操作员手填、"是否存在变更"由 PLM 按物料编码返回**，两者均不在 EBS 取数返回字段之列）；
- 单次返回是否有上限/分页（时间窗内物料多时会否被截断）。

**EBS — 回传"是否机加中心可以做"（挑图）**
- 接受字段、按行还是按批回传；⭐ **必传"是否机加中心可以做"(是/否)**，回传前客户端已做必填校验；
- ⭐ **回传按什么键关联回 EBS 原记录**（很可能是 **EBS-ID**，须确认 EBS-ID 是否即回传主键、回传时是否需带回，以及还需随行回传哪些字段）；
- ⭐ 幂等：重传是否会重复写；
- 成功/失败如何判断；是否接受只传正常行（异常行不传）。

**PLM — 按物料编码取图纸 + 物料数据**
- ⭐ **PLM 除图纸外还返回"是否存在变更"状态**（核价"当前是否存在变更"/挑图"是否存在变更"均取此，逻辑相同）：确认该字段的接口、取值/含义（布尔？枚举？变更单号？）、是否与图纸同一接口返回还是需另一接口；
- ⭐ 返回的是文件本体/下载地址，还是仅元信息（仅元信息则需第二个下载接口）；
- ⭐ **版本**：一个物料多版图纸，返回最新版/已发布版/全部？（通常要"当前有效版"）；因本地即数据、后端不留快照，须确认取到的就是建批次时的有效版；
- 一个物料多张图（PDF+STEP+以后格式）如何返回；大文件下载超时/大小限制。

**SRM — 回传核价**
- 接受字段、按行还是按批；⭐ **必传"目标价"（核价结果）**，回传前客户端已做必填校验；确认 SRM 侧字段映射（物料编码 / 目标价，及是否还需型号/名称/需求数量/是否变更等随行回传）；
- ⭐ 幂等：整批提交中途失败重试是否重复写（决定后端是否需记录"哪些行已成功、只补传失败的"）；
- 异常行完全不传，SRM 是否会因"缺几条"报错或对账不上；
- 成功/失败/部分失败如何返回。

**DHR（后续）**
- 认证协议（OIDC/LDAP/CAS/私有 API）；
- 提供的组织/岗位/工号字段是否够驱动"流程权限"与"工号取数"。

## 10. 部署形态（约 10 个操作员）

```
内网一台服务器(或 VM)
├─ ASP.NET Core 后端网关   (取数/回传 + 防腐层 + 认证 + 配置下发 + 回传审计日志，无状态)
└─ PostgreSQL              (配置 / 权限与工号映射 / 回传审计日志；不存业务数据与图纸)
        ▲ API（取数下发 / 回传）
   10 台 PC 上的 WPF 客户端  (ClickOnce 自动更新，内分核价/挑图功能区)
        └─ 本地文件系统：数据 + 图纸 + 状态（待处理/已处理 文件夹 + manifest）
           每个操作员固定用同一台机器
```

规模不大、内网部署，无需重型架构；后端 + 库可同机起步。**已无对象存储（MinIO 移除）**，图纸不在服务器集中存。
